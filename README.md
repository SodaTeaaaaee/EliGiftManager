# EliGiftManager

EliGiftManager is a Go + Wails desktop workspace for gift dispatch operations. The repository now contains both a backend data skeleton and a Vue 3 desktop shell, but the product is still in an exploratory phase.

## Current Status

- Backend groundwork exists: SQLite initialization, database models, CSV transformers, and validation services already live under `internal/`.
- Frontend groundwork exists: the desktop shell, sidebar navigation, and route-level pages for dashboard, orders, members, products, templates, and settings are in place.
- Wails bindings are intentionally wrapped behind `frontend/src/shared/lib/wails/app.ts` so UI code does not directly depend on generated bridge files.
- Important product caveat: business logic is **not finalized** yet, and the current UI is a **temporary prototype shell that will definitely be redesigned in a major way**.

## Stack

- Backend: Go + Wails + GORM + SQLite
- Frontend: Vue 3 SFC + TypeScript + Vue Router + Pinia
- Styling: Tailwind CSS, with selective Naive UI usage
- Tooling: Vite + Deno task runner
- Desktop shell: Wails asset pipeline and native window lifecycle

## Quick Start

```powershell
# install backend dependencies
go mod tidy

# install frontend dependencies through Deno
Set-Location frontend
deno install

# verify the frontend
deno task typecheck
deno task build

# return to the project root and start Wails development
Set-Location ..
wails dev

# run backend tests
go test ./...

# or build the packaged desktop binary
wails build
```

## Repository Notes

- `deno install` resolves npm dependencies and creates the compatibility `node_modules` directory required by Vite.
- `wails dev` uses the `frontend:dev:watcher` command declared in `wails.json` and connects to `http://127.0.0.1:5173`.
- `frontend/wailsjs` is generated by Wails and is committed because the frontend imports it.
- Direct `wailsjs` imports should stay inside `frontend/src/shared/lib/wails`.
- Current page content is still mostly mock data and layout scaffolding; do not treat the screen copy or interaction model as final product behavior.

## Structure

- `internal/config`: backend application metadata and window settings
- `internal/db`: SQLite initialization and migration entrypoints
- `internal/model`: database tables, template types, and domain enums
- `internal/service`: validation logic and CSV transformation services
- `frontend/src/app`: Vue app shell and router
- `frontend/src/pages`: route-level screens, including separate templates and settings pages
- `frontend/src/shared`: shared UI, types, and Wails wrappers
- `frontend/package.json`: Vite and Vue toolchain metadata
- `frontend/deno.json`: Deno task runner and npm compatibility settings
- `build`: Wails packaging assets
- `docs`: repository conventions and workflow notes

## Generated and Ignored Files

- `frontend/node_modules`: generated by `deno install`, ignored
- `frontend/package.json.md5`: generated by Deno npm compatibility bookkeeping, ignored
- `frontend/dist`: generated by Vite production builds, ignored
- `frontend/wailsjs`: generated by Wails bindings, committed
- `.cache`: local build and test caches, ignored
- `build/bin`: packaged binaries, ignored

See `docs/PROJECT-STRUCTURE.md` for the layout rationale and `docs/DEVELOPMENT.md` for the day-to-day workflow.
