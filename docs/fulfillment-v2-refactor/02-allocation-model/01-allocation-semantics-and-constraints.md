# 分配语义与约束

本文件聚焦会员权益与零售订单在分配语义上的根本差异，并明确当前 tag 系统在 V2 中应保留什么、约束什么。

### 3.1.4 分配语义的核心判断

V2 必须接受一个现实：

- 会员权益型需求和零售订单型需求，可以在最终履约层收敛
- 但不应该被强迫使用同一种“初始分配生成方式”

推荐明确区分两种基础语义：

1. `policy_driven`

特征：

- 上游给出的通常不是最终 SKU 履约行，而是身份、权益或资格
- 系统需要通过规则把“谁符合条件”推导成“该发什么”
- 允许在推导结果之上做协商式、补偿式、赠送式修正

适用：

- 会员权益型需求
- 活动回馈型需求
- 平台权威判定成立、但仍待本系统执行的阶段性会员礼物

2. `demand_driven`

特征：

- 上游给出的通常已经是明确需求行
- 系统优先尊重原始订单或原始需求行
- 后续可以调整，但调整应被视为对原始需求的显式修正，而不是重新推导原始需求

适用：

- 零售订单型需求
- 手工补单中带有明确商品行的需求
- 会员限定购买、支持者限定购买、会员专属折扣购买等“资格解锁交易”场景

补充判断规则：

- 如果不主动下单，系统也已经欠用户一个应发物，则更接近 `policy_driven`
- 如果不主动下单，就不会出现待履约商品行，则更接近 `demand_driven`

### 3.1.5 当前 tag 系统在 V2 中的定位

V2 不应否定当前 tag 系统的价值。

相反，应明确承认：

- 当前 tag 系统是一个优秀的 `policy_driven` 分配系统
- 它在会员权益场景下已经证明了高效、清晰、自由、易用

因此 V2 的原则不是“用一种全新的统一系统替换 tag”，而是：

1. 保留它在会员权益场景中的主导地位
2. 不强迫零售订单型需求先被翻译成 tag 再进入履约
3. 允许两种初始分配方式最终收敛到同一套 `FulfillmentLine`

### 3.1.6 会员体验不得退化

这是 V2 的强约束，而不是软建议。

当前 `policy_driven` 会员分配逻辑在以下方面已经具备明显优势：

- 商品级批量配置效率高
- 身份规则表达直接
- 例外覆盖灵活
- 创作者视角下非常易于理解

因此后续重构必须满足：

1. 会员权益型波次仍然可以使用规则驱动分配
2. 会员权益型波次的批量加赠、减赠、覆盖能力不得明显退化
3. 不能为了统一零售语义而破坏当前会员分配体验

换句话说：

- V2 应当扩展当前系统
- 而不是让会员分配 UX 变得比现在更笨重

### 3.1.7 有符号贡献、基础分配结果与最终履约结果不是同一层

当前旧实现里，负数 tag 已经在事实上承担了一部分“有符号规则贡献”的语义。

V2 更稳妥的做法，不是把这种语义简单理解成“最终履约行可以是负数”，而是明确拆开以下几层：

1. `Allocation Contribution`

- 表示某条规则对某位参与者、某件商品贡献了多少数量
- 可以是正数
- 也可以是负数

2. `Base Allocation Result`

- 表示初始分配层结算后的第一版履约结果
- 它应当是非负结果
- 如果规则贡献求和后小于等于 0，更适合解释为“不产生基础履约行”，而不是生成负数履约行

3. `FulfillmentAdjustment`

- 表示当前波次最终履约例外
- 它的 delta 也可以是正数或负数
- 但它仍然不是最终执行数量本身

4. `Resolved Fulfillment Result`

- 表示最终进入执行链的履约结果
- 更稳妥的规则应是：
  - `resolved = max(base + adjustment_delta, 0)`

这意味着：

- 负数在规则贡献层和调整 delta 层都可以合法存在
- 但最终进入执行链的履约结果不应为负
- “结果被压到 0”本身不应自动视为错误
- 是否需要复核，关键不在于出现负号，而在于是否出现语义错配、对象失效或外部 basis 错配

### 3.1.8 动态集合语义应留在 `Membership Allocation`

当前已确认：

- `Adjustment Review` 首版不应承担动态集合规则编辑
- 如果某个动作想表达的是“所有某类身份都一起变化”“某个集合未来继续自动跟随变化”
- 那它更接近初始分配规则，而不是波次内最终履约例外
- 当前阶段也不应默认同一位会员会带着复杂多身份交集进入系统
- 多身份交集 / 子集关系更接近上游平台语义，首版不应为此过早引入重型规则语言

因此更合理的分层是：

- `AllocationPolicyService`
  - 承载 selector、规则贡献、批量规则编辑语义
- `FulfillmentAdjustmentService`
  - 只承载具体参与者 / 具体履约行的最终例外语义
- `Membership Allocation`
  - 是 `AllocationPolicyService` 的主要 UX 入口
- `Adjustment Review`
  - 是 `FulfillmentAdjustmentService` 的主要 UX 入口

这里要特别强调：

- 页面不是领域所有者
- 页面只是工作流入口和观察视图
- 真正稳定的语义边界，应落在领域对象、应用服务和重算规则上

这里还要再补一条当前阶段的收敛约束：

- 现阶段不需要为了未来可能出现的“多身份交集 / 布尔表达式 / 复杂子集关系”而把规则语言做得过重
- 更稳妥的首版目标，是先覆盖当前真实会出现的简单 selector

首版更建议至少覆盖：

- 单平台身份等级
- `platform_all`
- `wave_all`
- 当前商品上的显式用户覆盖

同时在内部结构上保留未来扩展位：

- 如果未来真的出现平台侧复杂身份集合、交集或子集要求
- 应在 `Membership Allocation` 的规则层扩展
- 而不是把共享调整层演化成第二套规则引擎
