# WaveAllocationStep 演进方向

本文件整理 `WaveAllocationStep` 的正式命名、工作流定位和页面拆分原则，同时保留“会员分配 UX 不退化”这一强约束。

### 7.4 `WaveAllocationStep` 的正式定位

根据当前决策，文档层现在就正式采用：

- `WaveAllocationStep`

`WaveTagStep` 只在两种情况下继续出现：

- 讨论当前旧实现
- 说明从旧实现迁移到新概念的过程

换句话说，未来语义里不再把“tag step”当成总名，而把它视为 `WaveAllocationStep` 里偏 `policy-driven` 的一个遗留实现来源。

### 7.4.1 工作流分区优先于视觉分区

当前更合理的方向，不是把会员和零售硬塞进同一页面再做视觉分栏，而是：

- 把波次工作流拆成有顺序的大阶段
- 再把每个阶段拆成若干职责明确的小步骤
- 会员和零售在“初始分配生成”阶段走不同步骤和不同页面

这比单页视觉分区更稳健，原因是：

1. 会员与零售真正不同的，是初始语义和编辑心智，不只是页面布局
2. 会员用户不需要为了看零售映射表而接受新的认知负担
3. 零售订单也不需要被迫伪装成 tag 规则编辑问题
4. 混合波次仍然可以在后续共享同一套调整、校验、导出和回填链路

### 7.4.2 建议的大阶段与步骤

建议把波次编辑相关工作流组织成以下阶段：

1. `Demand Intake`
   - 导入和标准化上游需求
2. `Initial Allocation`
   - 生成第一版履约结果
3. `Adjustment Review`
   - 统一做修正、审查、异常确认
4. `Execution Readiness`
   - 地址、商品映射、缺项校验
5. `Supplier Execution`
   - 工厂导出与执行跟踪
6. `Shipment Intake`
   - 工厂发货回传
7. `Channel Sync / Closure`
   - 回填来源渠道或做人工闭环确认

其中 `Initial Allocation` 阶段建议拆成两个独立页面：

1. `Membership Allocation`
   - 面向 `policy-driven`
   - 主要服务会员权益型需求
   - 保留当前商品级批量规则编辑体验
2. `Demand Mapping`
   - 面向 `demand-driven`
   - 主要服务零售订单型需求
   - 核心目标是清晰展示原始需求行、商品映射、异常与缺失项

在这两个页面之后，再进入一个共享页面：

3. `Adjustment Review`
   - 面向所有波次
   - 统一承接加送、减送、替换、补发
   - 明确区分“原始需求真相”和“履约调整结果”

### 7.4.3 混合波次的实际含义

“混合波次”不应被理解为：

- 一个页面里同时混排两套截然不同的操作模型

更应该被理解为：

- 同一个 `Wave` 容器里，允许同时存在 `policy-driven` 和 `demand-driven` 的初始生成步骤
- 这些步骤写入同一套 `FulfillmentLine` 真相层
- 后续共享同一套调整、执行、发货、回填与闭环流程

也就是说，混合的是“履约容器”和“后链路”，不是“初始编辑心智”。

### 7.5 会员分配 UX 保留原则

当前 tag 系统在会员权益场景下已经证明是高效方案。

因此 `WaveAllocationStep` 的设计必须满足：

1. 会员权益型波次默认仍以 `Membership Allocation` 为主入口
2. 现有商品中心批量配置体验不得明显退化
3. “先规则，再覆盖，再预览”的认知路径应尽量保留
4. 零售订单型需求的引入不应迫使会员用户先理解订单映射语义
5. 共享的 `Adjustment Review` 页面不能反向污染会员分配主路径
