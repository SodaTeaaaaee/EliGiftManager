# WaveAllocationStep 演进方向

本文件整理 `WaveAllocationStep` 的正式命名、工作流定位和页面拆分原则，同时保留“会员分配 UX 不退化”这一强约束。

### 7.4 `WaveAllocationStep` 的正式定位

根据当前决策，文档层现在就正式采用：

- `WaveAllocationStep`

`WaveTagStep` 只在两种情况下继续出现：

- 讨论当前旧实现
- 说明从旧实现迁移到新概念的过程

换句话说，未来语义里不再把“tag step”当成总名，而把它视为 `WaveAllocationStep` 里偏 `policy-driven` 的一个遗留实现来源。

### 7.4.1 工作流分区优先于视觉分区

当前更合理的方向，不是把会员和零售硬塞进同一页面再做视觉分栏，而是：

- 把波次工作流拆成有顺序的大阶段
- 再把每个阶段拆成若干职责明确的小步骤
- 会员和零售在“初始分配生成”阶段走不同步骤和不同页面

这比单页视觉分区更稳健，原因是：

1. 会员与零售真正不同的，是初始语义和编辑心智，不只是页面布局
2. 会员用户不需要为了看零售映射表而接受新的认知负担
3. 零售订单也不需要被迫伪装成 tag 规则编辑问题
4. 混合波次仍然可以在后续共享同一套调整、校验、导出和回填链路

### 7.4.2 建议的大阶段与步骤

建议把波次编辑相关工作流组织成以下阶段：

1. `Demand Intake`
   - 导入和标准化上游需求
2. `Initial Allocation`
   - 生成第一版履约结果
3. `Wave Overview`
   - 波次内独立总览页
   - 采用只读优先原则
   - 承担阶段导航、聚合诊断、异常汇总与进入下一步判断
4. `Adjustment Review`
   - 统一做修正、审查、异常确认
5. `Execution Readiness`
   - 地址、商品映射、缺项校验
6. `Supplier Execution`
   - 工厂导出与执行跟踪
7. `Shipment Intake`
   - 工厂发货回传
8. `Channel Sync / Closure`
   - 回填来源渠道或做人工闭环确认

其中 `Initial Allocation` 阶段建议拆成两个独立页面：

1. `Membership Allocation`
   - 面向 `policy-driven`
   - 主要服务会员权益型需求
   - 保留当前商品级批量规则编辑体验
2. `Demand Mapping`
   - 面向 `demand-driven`
   - 主要服务零售订单型需求
   - 核心目标是清晰展示原始需求行、商品映射、异常与缺失项

在这两个页面之后，先进入一个波次内独立总览页，再进入共享编辑页：

3. `Wave Overview`
   - 面向所有波次
   - 用于统一查看聚合结果、异常、未纳入项统计和下一步阻塞
   - 与编辑层分离，以降低小窗口下的信息拥挤和心智混乱
   - 可作为进入 `Adjustment Review` 的显式关口，也可在无需编辑时直接放行到后续阶段
4. `Adjustment Review`
   - 面向所有波次
   - 统一承接加送、减送、替换、补发
   - 明确区分“原始需求真相”和“履约调整结果”

### 7.4.3 混合波次的实际含义

“混合波次”不应被理解为：

- 一个页面里同时混排两套截然不同的操作模型

更应该被理解为：

- 同一个 `Wave` 容器里，允许同时存在 `policy-driven` 和 `demand-driven` 的初始生成步骤
- 这些步骤写入同一套 `FulfillmentLine` 真相层
- 后续共享同一套调整、执行、发货、回填与闭环流程

也就是说，混合的是“履约容器”和“后链路”，不是“初始编辑心智”。

### 7.4.4 共享调整层的边界应由语义能力决定

当前已经形成的分流原则是：

- `Adjustment Review` 调整不了的内容，应返回前置页面处理
- `Adjustment Review` 能安全处理的内容，才适合收敛到共享调整层

但还有一层尚未收敛的判断：

- 某些动作虽然理论上两边都能做
- 仍需要基于操作者“想改的是上游真相，还是履约结果”来决定入口

这部分“意图分类”仍需后续继续细化。

### 7.4.5 共享调整层的当前工作定义

当前更稳妥的定义是：

- `Adjustment Review` 只处理当前波次内的最终履约例外
- 不处理上游需求真相修正
- 不处理默认生成逻辑修正
- 不处理动态集合规则修正

可以先用三个问题做分流：

1. 你想改的是“上游原始事实”吗

- 如果是，应返回前置页面或上游导入侧处理

2. 你想改的是“系统默认怎么生成第一版结果”吗

- 如果是，应返回 `Membership Allocation` 或 `Demand Mapping`

3. 你想改的是“这次波次最终到底发什么”吗

- 如果是，才适合进入 `Adjustment Review`

这里还要再加一条更细的判断：

4. 你想改的是“已经选中的具体对象”，还是“某个身份 / 平台 / 集合以后都应这样变化”？

- 如果是前者，才适合继续进入 `Adjustment Review`
- 如果是后者，应返回 `Membership Allocation`

### 7.4.6 `Adjustment Review` 首版允许动作清单

首版建议允许以下动作：

- 对当前波次某条最终履约行做加送
- 对当前波次某条最终履约行做减送
- 对当前波次某条最终履约行做替换
- 对当前波次某条最终履约行做移除
- 对当前波次某位参与者增加一条显式补发 / 补偿型履约项
- 对最终履约结果做波次内的一次性数量修正

这些动作的共同特征是：

- 它们表达的是本波次的执行例外
- 它们不声称上游需求本来就是这样
- 它们不自动改写未来同类需求的默认生成逻辑
- 它们的 target 应是明确的具体参与者或具体履约行
- 而不是一个会随重算继续扩张或收缩的动态 selector

### 7.4.7 `Adjustment Review` 首版禁止动作清单

首版不应允许以下动作直接在共享调整层完成：

- 改写某条 `DemandLine` 是否成立
- 改写上游订单原始数量、原始 SKU、原始订单号
- 改写 `RecipientInputState`
- 改写 `RoutingDisposition`
- 改写 `EntitlementAuthority`
- 改写会员规则本身
- 改写零售映射规则本身
- 直接新增一个当前波次里本来不存在的全新参与者
- 把应当批量生效的默认逻辑修正伪装成多条逐行例外
- 基于身份 / 平台 / 交集 selector 在共享调整层里做规则式批量动态改动

这些动作之所以禁止，是因为它们改动的不是“本波次最终发什么”，而是：

- 上游真相
- 接手边界
- 初始生成规则

### 7.4.8 灰区动作的当前判断规则

确实存在一些动作，表面看起来两边都能做。

当前建议先按下面的工作规则处理：

- 如果只想修正当前波次、当前少量对象的最终结果，优先进入 `Adjustment Review`
- 如果希望同类对象在本波次大范围一起改变，优先返回前置页面修正规则或映射
- 如果希望未来波次也按新逻辑生成，必须返回前置页面修正规则或映射
- 如果希望某类 selector 在本波次内继续动态跟随后续重算，也应返回前置页面

换句话说：

- `Adjustment Review` 面向 `wave-local exception`
- 前置页面面向 `default generation logic`

### 7.4.9 当前阶段不把共享调整层当作手工终表编辑器

根据当前决策：

- 共享调整层不是“随便补一行最终导出内容”的地方
- 更不是一个手工拼装终表的兜底页面

如果需要新增一个本来不在波次里的对象：

- 更合理的入口应是导入侧、手工接手侧，或前置参与者整理阶段
- 而不是绕过前置结构直接塞进 `Adjustment Review`

### 7.5 会员分配 UX 保留原则

当前 tag 系统在会员权益场景下已经证明是高效方案。

因此 `WaveAllocationStep` 的设计必须满足：

1. 会员权益型波次默认仍以 `Membership Allocation` 为主入口
2. 现有商品中心批量配置体验不得明显退化
3. “先规则，再覆盖，再预览”的认知路径应尽量保留
4. 零售订单型需求的引入不应迫使会员用户先理解订单映射语义
5. 共享的 `Adjustment Review` 页面不能反向污染会员分配主路径
6. 波次内跨步骤导航不应破坏会员分配已经形成的规则结果和例外覆盖
