# 决策记录与继续讨论事项

本文件不再只记录“待决策问题”，而是同时记录当前已经形成的决策和仍需继续讨论的点。

## 已确认的决策

### 1. 正式采用 `WaveAllocationStep`

已确认：

- 文档层现在就正式采用 `WaveAllocationStep`
- `WaveTagStep` 只保留为旧实现语境下的历史名字

原因：

- 尽早统一命名，能减少语义混淆
- 后续代码迁移可以分阶段做，但文档和讨论不应继续混用两个总名

### 2. 混合波次按工作流分区，不按单页视觉分区

已确认：

- 混合波次需要分区
- 这个分区首先是工作流分区，不是把两套语义强行放进一个页面做视觉切块
- 工作流可以拆成有顺序的大阶段
- 每个大阶段下可以再拆成若干没有强制先后关系的小步骤
- 会员和零售在首版中应当是两个独立步骤、两个独立页面

当前文档已据此更新为：

- `Membership Allocation`
- `Demand Mapping`
- `Wave Overview`
- `Adjustment Review`

其中前两者是不同的初始分配入口，第三者是波次内独立总览页，最后一者是共享收敛层。

### 3. 零售渠道允许进入共享调整层

已确认：

- 零售订单允许进入和会员一致的 `Adjustment Review` 共享调整层
- 但必须严格区分：
  - 原始需求真相
  - 履约调整结果

这意味着：

- 允许“加送 / 减送 / 替换 / 补发”
- 但这些行为不能覆盖或伪装原始订单语义

### 4. 模板系统直接升级为 Profile 系统入口

已确认：

- V2 首版采用直接升级路线
- 不再把旧模板系统当成需要长期并行保留的成熟入口

原因：

- 旧模板系统能力边界本来就不清
- 与特定测试数据耦合偏深
- 目前更需要先把核心语义和能力边界定清楚

### 5. 轻量人工闭环只能记录决策，不能覆盖事实状态

已确认：

- 人工进度只能记录闭环决策
- 不能覆盖工厂发货、物流存在、来源渠道同步这些事实状态

同时已确认：

- 这类人工决策应进入统计
- 但要单独归类

### 6. 支持关系导出的实体礼物默认仍归 `membership_entitlement`

已确认：

- Patreon membership merch、FANBOX support plan 礼物、直播平台会员礼物
- 在当前认知下都不需要单独拆出新的顶层 `demand_kind`

更稳妥的做法是：

- 保留 `membership_entitlement`
- 通过 `ObligationTriggerKind`、`EntitlementAuthority`、`RecipientInputState` 区分其内部差异

### 7. 会员限定购买应视为 `retail_order` 的变体

已确认：

- 当会员或支持关系只提供购买资格，而履约义务必须在后续主动下单时才成立
- 该需求应归类为 `retail_order`

补充语义：

- 可通过 `eligibility_context_ref` 保留其资格来源
- 可通过 `ObligationTriggerKind = supporter_only_purchase` 表达其触发方式

### 8. 平台权威成就应优先交由上游平台判定

已确认：

- 对连续订阅阶段性礼物、earned merch 等场景
- 如果平台自己掌握完整历史并定义了达成规则
- 本系统不应在缺失历史的情况下自行重算资格

推荐语义：

- `EntitlementAuthority = upstream_platform`
- 本系统主要负责输入采集、路由、履约执行与闭环

### 9. 共享总览页采用方案 B

已确认：

- 共享总览页需要存在
- 采用方案 B
- 它是波次内独立页面
- 它不与 `Adjustment Review` 合并

原因：

- 当前窗口空间有限，强行把聚合浏览与复杂编辑压进同一页，容易拥挤和混乱
- 混合波次需要一个明确的会合点，用来统一查看聚合结果、异常与待处理项
- 该页更适合承担阶段导航、聚合诊断和进入下一步判断，而不是复杂编辑

### 10. `Wave Overview` 采用只读优先原则

已确认：

- `Wave Overview` 应优先保持语义干净、工作流直观
- 编辑职责原则上放在统一的 `Adjustment Review`
- `Wave Overview` 不应自然滑向“又一个编辑页面”

当前方向：

- 首版按只读优先设计
- 它主要负责观察、汇总、诊断、导航、决定下一步
- 若后续确实引入轻量动作，也应满足“不会让用户误解它是编辑层”这一约束

### 11. 波次编辑需要更强的步骤向导与跨步骤导航

已确认：

- 波次内需要一个更强的步骤向导
- 用户应能在任意步骤之间快速跳转
- 形式可以是可点击流程图或其他清晰导航 UI

这不是纯前端问题，同时意味着：

- 后端和数据结构必须支持跨步骤往返编辑
- 不应因为跳回前置页面而大幅破坏后续页面已经形成的数据

### 12. 非破坏性编辑是波次编辑的核心设计原则之一

已确认：

- 在整个波次编辑过程中，不同步骤更像是对同一批业务对象的不同视角
- 页面之间可以双向往返
- 页面切换不应隐式重写、清空或破坏其他步骤已经确认的数据

这意味着：

- `Membership Allocation`、`Demand Mapping`、`Wave Overview`、`Adjustment Review` 需要共享同一套稳定底层真相
- 不同步骤应尽量只修改自己拥有的那一层语义
- 任何跨层重算都需要可解释、可预期、可审计

### 13. `Wave Overview` 可以作为进入共享调整层的显式关口

已确认：

- `Wave Overview` 可以承担进入 `Adjustment Review` 的显式关口职责
- 但它不是强制的“所有波次都必须编辑后才能继续”的卡点

更准确地说：

- 用户可以在 `Wave Overview` 决定“进入统一调整层继续编辑”
- 也可以在确认无需进一步编辑后，直接进入后续执行准备阶段

### 14. 采用“三问分流”作为当前工作规则

已确认：

- 当操作者要改“上游真相”时，应返回前置页面或导入侧
- 当操作者要改“默认生成逻辑”时，应返回 `Membership Allocation` 或 `Demand Mapping`
- 当操作者要改“本波次最终履约例外”时，才进入 `Adjustment Review`

这个规则的价值是：

- 把抽象的“意图”先收敛成可执行分流问题
- 让 `Wave Overview` 能按“下一步该去哪里”而不是只按异常类型组织信息

### 15. `Adjustment Review` 首版只处理波次内最终履约例外

已确认：

- 首版 `Adjustment Review` 只处理 `wave-local final fulfillment exception`
- 不承担上游需求真相修正
- 不承担默认生成逻辑修正

首版可允许：

- 加送
- 减送
- 替换
- 移除
- 补发 / 补偿
- 当前波次的一次性数量修正

首版不应允许：

- 改写原始订单真相
- 改写权益成立与否
- 改写 `RoutingDisposition`
- 改写 `RecipientInputState`
- 改写规则本身
- 改写默认映射本身

### 16. `Adjustment Review` 首版不允许直接新增全新参与者

已确认：

- 共享调整层不应直接新增一个本来不在当前波次里的全新参与者
- 首版共享调整层只修改已经进入当前波次处理范围的对象
- 对这些对象增加补发 / 补偿型履约项仍然允许

原因：

- 如果在共享调整层里直接新增全新参与者，语义上就过于接近“手改最终导出表格”
- 这样既享受不到前置分配规则与映射系统的便利，也会冲淡工作流边界
- 如确有手动新增对象需求，也更适合在导入/接手阶段完成，再进入后续共享链路

### 17. 当前阶段不为旧数据兼容和过渡映射背负额外约束

已确认：

- 当前软件仍处于早期开发阶段
- 不需要为旧数据库状态、旧覆盖数据、旧波次结果设计额外的过渡兼容方案
- 如果某个问题在删库从零开始后自然消失，可以视为当前阶段不存在

这意味着：

- 文档和实现可以直接按目标语义收敛
- 不必为了尚未投产的旧数据形态保留复杂迁移包袱
- 这里的“过渡”主要指开发实现顺序，不指生产数据双轨兼容
- 对于已经明确的新语义，如果没有历史包袱，应尽量一次到位

### 18. 工厂提交后仍允许继续编辑，并以当前工作区结果为准

已确认：

- 任何时候、任何状态，都允许继续修改内容
- 软件的目标是辅助履约，而不是限制操作者
- 提交工厂后的改动，原则上以覆盖当前工作区结果为主

必须同时明确：

- 这不代表系统完全不需要保存工厂导出、物流回传、来源渠道回填这些对象
- 这些对象仍然需要存在，用于当前工作区的映射、导入导出和状态辅助
- 但它们不应被抬升成“不可更改的历史真相账本”

### 19. 动态集合规则属于 `Membership Allocation`，不属于 `Adjustment Review`

已确认：

- 如果某个动作想表达“某类身份 / 某个平台 / 某个集合以后都应这样变化”
- 那它属于规则层
- 首版不应放进 `Adjustment Review`

这意味着：

- `Adjustment Review` 首版不支持基于身份 / 平台 / 交集 selector 的动态批量规则编辑
- 这类能力如果未来要扩展，也应继续留在 `Membership Allocation`
- 当前阶段也不应默认同一位会员会携带复杂多身份交集进入系统
- 现阶段更稳妥的范围，仍是单平台身份等级、`platform_all`、`wave_all` 和显式对象覆盖

### 20. 有符号贡献可以存在，但基础结果与最终结果不应为负

已确认：

- 规则贡献层允许正负数量
- 共享调整层 delta 允许正负数量
- 但基础分配结果与最终进入执行链的履约结果不应为负

更准确地说：

- 负号更适合存在于 `AllocationContribution` 或 `FulfillmentAdjustment.quantity_delta`
- 最终执行结果更接近：
  - `resolved = max(base + delta, 0)`

因此：

- “结果被压到 0”本身不自动等于错误
- 是否复核，关键看 target/basis 是否 still valid，而不是单纯看见负号

### 21. `Adjustment Review` 首版只处理具体对象例外，不处理 selector 级批量目标

已确认：

- 共享调整层的 target 应是具体参与者、具体履约行，或明确展开后的具体对象
- 首版不应让共享调整层直接持有会随重算继续扩张 / 收缩的动态 selector 目标

这也意味着：

- 首版共享调整层可以继续处理具体对象的加送 / 减送 / 替换 / 补发
- 但不应反向长成第二套动态规则引擎

### 22. 撤销 / 重做采用持久化树状工作区历史

已确认：

- 不是线性撤销栈
- 撤销到旧节点后重新编辑，旧未来分支必须保留
- 软件关闭后重新打开，历史仍应存在

更准确地说：

- 默认交互可以看起来像线性历史
- 但底层必须保留树状分支结构
- 只有在高级历史图里，用户才需要显式切换分支

### 23. 撤销 / 重做只回滚本地工作区，不回滚外部动作

已确认：

- 普通 undo / redo 只作用于本地工作区历史
- 不应伪装成真的撤销了工厂导出、物流导入或来源渠道同步

因此：

- 外部对象仍需保留
- 它们与当前工作区如果脱节，应进入 basis 偏离或复核提示

### 24. 全应用共用同一套 history 基础设施，但优先完善 `wave`

已确认：

- history infra 不应只属于波次页
- `templates / products` 等页面也应逐步纳入
- 但优先级上先把 `wave` scope 做稳

### 25. Undo / Redo 必须有即时可感知反馈和短期可回看入口

已确认：

- 每次撤销 / 重做后，用户都应收到显眼但不碍事的提示
- 提示应自动消失
- 但短时间内仍应可以翻出查看最近动作

当前更稳妥的结构是：

- 即时 toast
- 短期回执托盘

### 26. 全局快捷键不能劫持文本输入自身的原生撤销语义

已确认：

- 当焦点位于 `input / textarea / contenteditable` 时
- `Ctrl+Z / Ctrl+Shift+Z` 应优先服务文本输入自身
- 只有非文本输入上下文时，才更适合触发工作区级 history

### 27. 命名应尽早收敛，旧名只作过渡桥接

已确认：

- 当前没有历史包袱，因此新语义不应继续拖延命名
- 文档、service、DTO、页面、测试名和新 API 应尽量直接使用目标业务名
- 旧名可以短暂作为 compatibility alias，但不应长期和新名并列存在
- 真正需要分阶段处理的是物理迁移成本，而不是语义命名本身

这意味着：

- 如果一个概念已经被完整定义，就应尽快统一新名
- 不要先占位旧名，再在未来某个未知时点慢慢换
- 越早统一，越不容易让实现、文档和交互同时长出双重语义

## 仍需继续讨论的问题

### 1. 灰区动作的归属规则是否还需要更细的子分类

当前已确认的主规则已经足够支撑首版：

- 改上游真相，回前置层
- 改默认逻辑，回前置层
- 改当前波次最终例外，进共享调整层

当前已经进一步确认的是：

- 批量例外只有在逻辑上确实构成一个合理“单一例外语义”时，才适合留在共享调整层
- 如果某类“先例外、后固化”的动作频繁出现，更可能意味着前置分配系统能力不足
- 对危险操作应有提示文案或二次确认，但应保持适量，不能破坏便捷易用原则

但仍未定义清楚的是：

- 如何把“合理批量例外”和“其实应回前置层的批量默认逻辑修正”做出更可操作的界线
- 哪些危险操作应算轻提示，哪些应算强确认

### 2. `Wave Overview` 首版应展示哪些聚合视角

当前倾向：

- 不预设一次性完美的信息架构
- 先基于若干明确假设做首版落地
- 再通过实际使用反馈迭代调整

后续需要继续细化：

- 首版最小可用分组是什么
- 哪些指标必须首版就有
- 哪些视角可以推迟到真实使用后再补

当前已经得到的明确授权是：

- 首版最小视角可以先由当前工作流推导决定
- 后续再纳入敏捷反馈循环迭代调整

### 3. 调整层重放与前置层重算的精确顺序如何落库

当前方向已经明确：

- 前置层先重建基础履约结果
- 共享调整层再显式重放

当前已经明确的是：

- 当前阶段不存在旧数据兼容与过渡映射负担
- 工作流应尽量接近 Blender modifier 的思路：
  - 先重建基础结果
  - 再重放共享调整层

但仍需继续讨论的是：

- 在什么条件下，应提示“当前基础变化会让既有例外失配，需要复核”

当前已经进一步确认的是：

- 负数贡献或负 delta 本身不应自动进入复核
- 最终结果被压到 0，也可能只是合理业务语义

更适合触发复核的，是：

- 共享调整层引用的 target 已不存在
- 共享调整层 target 仍存在，但引用语义变得歧义
- 最近一次导出 / 物流 / 回填 basis 已无法无歧义映射到当前对象

仍需继续细化的是：

- 哪些情况属于弱提示
- 哪些情况必须升级成强复核
- review 分类在 `Wave Overview` 中如何聚合展示

### 4. 导出/回传对象与当前工作区结果脱节时，系统该如何提示

当前已经确认：

- 工厂提交后仍允许继续编辑
- 软件以当前工作区结果为准
- 系统不是不可更改的历史账本

但还需要继续细化：

- 当用户修改了已经导出过的履约结果后，系统是否应显示“当前结果已偏离最近一次导出快照”
- 当用户在导入物流回传后又修改了履约结果，系统是否应显示“当前结果已偏离最近一次物流映射基础”
- 这些提示应是弱提示、强提示，还是只出现在 `Wave Overview`

注意：

- 这不是纯 UI 问题
- 首先要定的是“是否显式建模偏离信号”以及“偏离信号是否要求复核”
- 交互形式和文案可以后置给反馈循环，但语义本身不能悬空

当前已经进一步确认的是：

- 如果后续修改会导致已传出的对象信息出现错配，应提示复核
- “冻结旧版本”更适合冻结当时的 basis 引用与必要投影
- 不必默认备份整个波次的全部旧状态

更准确地说：

- `SupplierOrder / Shipment / ChannelSyncJob` 应保留各自 basis history node 与 projection snapshot
- 当前工作区后续变化时，再比较 basis 是否 still valid

仍需继续细化的是：

- 导出 basis、物流 basis、回填 basis 的提示强度分级
- basis 冻结 / 丢弃 / 重新生成的具体交互文案与入口

### 5. 哪些问题可以交给敏捷反馈循环，哪些不可以

可以交给反馈循环的问题：

- `Wave Overview` 首版分组命名是否还要微调
- 阶段导航采用什么视觉样式
- 哪些计数放首屏、哪些放次屏
- 哪些提示文案更清晰
- 哪些异常桶名称更自然

不能只靠反馈循环解决的问题：

- 当前工作区与最近一次导出、物流回传、渠道回填基础的偏离是否要被显式建模
- 偏离信号应当如何命名、归类和触发
- 偏离后是弱提示、强提示，还是进入复核
- 前置层重算后，已确认的共享调整层例外如何判定失配

结论：

- 反馈循环适合改“看法”
- 不适合替代“语义边界”

### 6. 工作区历史的保留策略与反馈时长还需要定多细

当前已经确认：

- 历史必须持久化
- 分支不能丢
- toast 与短期回执需要存在

但仍未完全定死的，是较低优先级的参数问题：

- checkpoint 多久打一次更合适
- 最近多少 node 需要完整保留
- 未 pin 分支的 GC 触发条件
- toast 停留时长
- 短期回执托盘保留多久、保留多少条

这类问题的重要性低于语义边界本身。

当前更适合的策略是：

- 先把“必须持久化、必须保留分支、必须有回执”定清
- 再在实现和试用阶段调整具体阈值
