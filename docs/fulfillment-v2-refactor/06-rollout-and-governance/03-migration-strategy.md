# Greenfield 重建与切换策略

本文件定义 V2 的 greenfield 重建总原则、保留边界与切换方式。

## 12. Greenfield 重建策略

### 12.1 总原则

当前这份 V2 计划已经明确采用：

- greenfield 重建

因此这里的重点不是：

- 如何让旧业务代码继续演进
- 如何让旧数据库结构长期并存
- 如何设计面向旧实现的兼容层

而是：

- 如何保留必要文档与考古资料
- 如何清理旧业务代码与技术债惯性
- 如何让新版本从第一天就按目标语义和设计模式落地

### 12.2 保留边界

应保留：

- `docs/`
- 备份分支
- 仓库历史
- 必要的工程壳子与构建配置
- 必要的开发工具链与环境约定

这里的“保留”有一条更严格的约束：

- 只有当某部分工程壳子是中性的、不会把旧业务结构和旧心智惯性带进新版本时，才保留
- 只要某段代码已经明显带有旧项目的业务倾向或架构惯性，就应删除后重写

不应保留为连续演进基座的内容：

- 旧业务 controller / service / model 实现
- 旧状态模型
- 旧模板入口
- 旧页面流程
- 为旧结构存在的桥接逻辑
- 带有旧项目业务倾向的“半中性工程代码”

### 12.2.1 工程壳子的判断标准

当前更稳妥的判断标准是：

可以保留的典型内容：

- 桌面应用启动入口
- 构建配置
- 开发脚本
- 数据目录约定
- 生成链路
- 与业务无关的最小运行时包装

只有当它们满足以下条件时才保留：

- 不依赖旧业务对象命名
- 不偷偷固化旧页面流程
- 不把旧数据结构假定写进初始化逻辑
- 不要求新版本继续沿用旧 controller/service/module 切分

一旦不满足，就应直接删掉重来。

### 12.3 切换方式

更稳妥的切换方式是：

1. 先冻结文档与基线
2. 明确保留资产
3. 删除旧业务代码
4. 建立新代码骨架
5. 从第一个可执行纵切面开始重建

这里的“切换”更接近：

- 同仓库内重建新版本

而不是：

- 在旧代码旁边长期并排维护两套业务实现

### 12.4 数据与 Schema 原则

当前阶段默认前提仍然是：

- 可以删库重来
- 可以重建开发环境
- 没有必须长期保留的生产旧包袱

因此更稳妥的原则是：

- 新版本直接建立目标 Schema
- 不为旧表补过渡字段
- 不为旧数据库形态设计长期兼容层

如果未来真的出现生产数据包袱：

- 再基于当时的真实状态单独设计导入或迁移方案

而不是提前为假设包袱牺牲当前架构清晰度。

### 12.5 工作区历史与 Basis 的起点

当前已经确认：

- 撤销 / 重做历史需要持久化
- 但不需要为 V2 之前的旧版本补造完整历史

因此更稳妥的策略是：

- `HistoryScope / HistoryNode / HistoryCheckpoint / HistoryPin`
  从 V2 新路径开始产生
- `SupplierOrder / Shipment / ChannelSyncJob` 的 basis 引用
  也从 V2 新对象开始记录

这意味着：

- 旧版本历史不回填
- 旧对象 basis 链不补造
- 新版本从第一批新对象开始具备完整语义

### 12.6 第一批重建目标应是可执行纵切面

greenfield 实施下，最危险的错误之一是：

- 先把所有目录和所有对象空搭一遍
- 但没有一个能跑通的纵切面

因此更稳妥的策略是：

- 先选一条最关键的端到端链路
- 用新模型把它完整跑通
- 再扩展到 shipment、channel sync、history、profile 等后续层

推荐默认首条纵切面：

- `Demand Intake -> Initial Allocation -> Wave Overview -> SupplierOrder Export`

原因：

- 它能最早验证上游需求、初始策略、波次容器、执行真相和工厂导出的主干结构
- shipment / channel sync / history 都可以在此基础上继续扩展

这里还要明确：

- “第一条端到端纵切面”
- 本质上就是旧代码删除后的第一块正式施工区域

但它不是：

- 只搭一个目录
- 只做一个页面
- 只建一张表

它必须是一条可以真正跑通的最小主干链路：

- 有输入
- 有核心领域处理
- 有用户可见输出

因此它会直接决定：

- 第一批目录骨架
- 第一批 service / projection / connector 边界
- 第一批数据库表
- 第一批页面
