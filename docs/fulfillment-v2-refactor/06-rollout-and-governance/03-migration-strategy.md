# 数据迁移策略

本文件定义 V2 的迁移总原则、开发实施分期与建议落地方式。

## 12. 数据迁移策略

### 12.1 总原则

迁移必须做到：

- 可重复
- 可回滚
- 不破坏现有有效数据
- 如果真的存在旧生产数据，再考虑兼容路径

当前这份 V2 计划默认的前提是：

- 可以删库重来
- 可以重建开发环境
- 没有必须长期保留的生产旧包袱

所以这里的“迁移”主要指：

- 代码结构迁移
- 领域语义迁移
- 新旧实现切换
- 开发期间的最小风险分阶段推进

不是指：

- 为旧生产数据设计复杂兼容层
- 为历史数据包袱长期维持双语义双结构

### 12.2 建议策略

1. 先新增表，不删旧表
2. 先新增字段，不强制重命名旧字段
3. 先让新功能写新表，再考虑是否回写旧表
4. 旧页面继续读旧投影，新页面或新功能读新投影

### 12.3 开发过渡建议

开发过渡阶段内建议：

- 如果某个旧实现还在帮助当前开发路径稳定运行，可以临时保留
- 但只要新语义已经定清，就应尽量一次到位切到新名、新表或新接口
- 不应为了“以后可能兼容”而故意保留双轨命名或双轨语义

最终目标：

- 新状态由多维字段和聚合投影驱动

### 12.4 工作区历史迁移建议

当前已经确认：

- 撤销 / 重做历史需要持久化
- 但不需要为旧版本未存在的历史做复杂回填

因此更稳妥的迁移策略是：

1. 新增 history 相关表

- `history_scopes`
- `history_nodes`
- `history_checkpoints`
- `history_pins`

2. 不回填旧版本工作区历史

- V2 上线前的旧编辑过程不必追溯重建
- 新历史从 V2 新路径开始产生即可

3. basis 引用只对新对象生效

- 新创建的 `SupplierOrder / Shipment / ChannelSyncJob` 才写入 `basis_history_node_id` 与 projection hash
- 不要求为旧对象补造完整 basis 链

4. 允许一段时间内“新旧页面不同步支持历史”

- `wave` 工作区优先接入
- 其他页面可以稍后按 scope 逐步纳入

这样能避免：

- 为尚未投产的旧数据背上高成本历史回填包袱
- 把 history 功能和旧路径实现耦合得过深

补充说明：

- 这里的“分段接入”是实现顺序，不是语义妥协
- 一旦某个 scope 已经定型，相关命名和结构应尽快一次到位
