# 风险与当前结论

本文件整理当前已识别的主要风险，并保留本轮规划阶段的核心结论。

## 14. 风险与注意事项

### 14.1 最大风险：在旧术语上继续硬补

如果继续把：

- `Member` 当成唯一客体
- `DispatchRecord` 当成唯一流程真相
- `exported` 当成流程终点

那么后续每接一个平台都会继续堆特例，最终很难维护。

### 14.2 第二大风险：过早重命名数据库表

如果一开始就强行把旧表重命名成新语义，迁移复杂度和风险都会显著提升。

建议：

- 先重构语义和读写路径
- 再决定是否物理重命名表

### 14.3 第三大风险：用单一进度条伪装复杂流程

V2 后，单一百分比会越来越不真实。

建议：

- 前端以阶段 + 计数 + 风险提示为主
- 百分比只作为辅助投影，不作为唯一信息来源

### 14.4 第四大风险：强迫所有需求都走 tag 规则系统

当前 tag 系统在会员权益场景下表现优秀，但如果强迫零售订单型需求也全部先转换成 tag，再生成履约行，会带来以下风险：

- 冲淡原始订单行语义
- 让零售真相来源变得难以审计
- 为零售场景引入不必要的规则复杂度
- 反过来污染会员场景的原有优秀体验

建议：

- 保留 tag 作为 `policy-driven` 分配主引擎
- 不把它当作所有需求类型的统一真相入口
- 让零售订单主要走 `demand-driven` 直入路径

### 14.5 第五大风险：为适配零售语义而让会员分配 UX 退化

V2 最大的产品风险之一，不是“支持不了零售”，而是“支持零售之后把会员分配做笨了”。

建议：

- 会员分配体验作为独立约束保留
- `WaveAllocationStep` 中默认保留规则驱动主入口
- 新增的 demand / adjustment 能力不应迫使会员用户改用更复杂的认知路径

### 14.6 第六大风险：把辅助系统误做成系统外履约总账

如果让系统继续膨胀成：

- 记录所有系统外履约事实
- 假定自己知道所有线下处理结果
- 用一个字段同时表达“本系统不接手”和“外部已履约完成”

那么系统边界会迅速失真。

建议：

- EliGiftManager 只记录“是否纳入本系统处理、为什么不纳入、何时可纳入”
- 不默认承担系统外履约完成事实的权威记录职责
- 使用 `RoutingDisposition` 表达路由决策，而不是伪造系统外履约状态

### 14.7 第七大风险：在缺乏完整历史时本地重算平台权威成就

对连续订阅阶段性礼物、earned merch 这类场景，如果平台自己定义了达成规则并掌握完整支付历史，而本系统又尝试在缺失月份、缺失手工履约记录的情况下本地重算，就会产生严重失真。

建议：

- 当平台掌握完整权威判定时，优先使用 `EntitlementAuthority = upstream_platform`
- 本系统主要承接导入后的领取参数采集、路由、履约执行与闭环

### 14.8 第八大风险：跨步骤跳转伴随隐式重写，破坏已有编辑结果

如果未来波次流程支持更强的步骤导航，但底层数据模型仍然要求：

- 回到前置页面就整体重算
- 重新映射就覆盖共享调整层结果
- 切换视图就清空另一页面已经确认的例外

那么最终用户虽然看起来“能跳转”，实际却不敢跳。

建议：

- 把非破坏性编辑明确为核心约束
- 让不同时步骤尽量只修改自己拥有的语义层
- 任何跨层重算都必须可解释、可预期、可审计

### 14.9 第九大风险：试图一次性设计完美的 `Wave Overview`

`Wave Overview` 的价值很大，但如果一开始就试图把所有聚合视角、诊断维度和导航逻辑一次性做完，风险也很高。

建议：

- 首版采用假设驱动方式落地
- 先交付最小可用的聚合与分流能力
- 再根据真实使用反馈迭代收敛信息架构

### 14.10 第十大风险：把辅助工作区对象误做成不可覆盖的历史账本

如果继续把：

- 工厂导出对象
- 物流回传对象
- 回填任务对象

都直接当成不可改写的历史真相，那么很快会和当前产品定位冲突。

当前定位更接近：

- 辅助履约工作区
- 当前状态协调器
- 导入导出与映射工具

而不是：

- 权威执行历史总账

建议：

- 保留这些对象，因为它们对当前工作流仍然必要
- 但把它们理解为当前工作区里的最新已知辅助对象
- 对“当前结果已偏离最近一次导出/回传基础”的情况，优先用辅助提示表达，而不是用硬锁表达

### 14.11 第十一个风险：把仍需先定清的语义问题误交给反馈循环

反馈循环适合修正：

- `Wave Overview` 的分组顺序
- 指标布局
- 文案措辞
- 阶段导航样式

但不适合替代：

- 偏离信号是否存在
- 偏离信号如何计算
- 偏离后是否需要复核
- 前置重算后例外如何判定失配

如果这些问题过早交给 UI 反馈，就很容易把语义边界做成不断漂移的交互债。

### 14.12 第十二个风险：让 `Adjustment Review` 重新长成第二套动态规则引擎

如果因为共享调整层看起来“方便集中处理”，就继续把：

- 身份 selector
- 平台 selector
- 交集 / 子集集合
- 动态跟随后续重算的规则

都塞回 `Adjustment Review`，那么很快会带来两个问题：

- 前置规则层与共享调整层边界再次变脏
- 共享调整层重新长出第二套 `AllocationPolicyRule`

建议：

- 动态 selector 语义严格留在 `Membership Allocation`
- 共享调整层只处理具体对象例外

### 14.13 第十三个风险：用线性 undo 栈破坏分支保留能力

如果撤销到旧节点后重新编辑，就把旧未来直接删掉，那么：

- 用户会失去探索不同方案的自由
- history 无法和 basis 引用稳健协同
- 一旦误操作，很难恢复到之前的另一条路径

建议：

- 必须保留树状分支
- 默认 redo 仍可沿首选分支前进
- 只有在高级历史图里才显式暴露分支切换

### 14.14 第十四个风险：用“每步完整快照”粗暴实现树状历史

虽然完整快照最直观，但如果每一步都保存：

- 整个 wave
- 全部规则
- 全部履约行
- 全部状态投影

那么很容易对普通设备造成明显的存储和内存压力。

建议：

- 日常以 patch 为主
- 周期性打 checkpoint
- basis 节点 pin
- 老旧未 pin 分支做压缩或 GC

### 14.15 第十五个风险：全局快捷键劫持文本输入的原生撤销

如果在文本输入框里按 `Ctrl+Z` 也直接触发工作区历史：

- 表单录入体验会非常糟糕
- 用户会对全局 undo/redo 产生强烈不信任

建议：

- 文本输入上下文优先使用原生 undo/redo
- 只有非文本编辑上下文时，再触发工作区级快捷键

---

## 15. 当前结论

本次重构的核心不是“加一个快递单号字段”，也不是“把波次状态多加几个枚举”。

真正要做的是：

- 把 EliGiftManager 从“短链路导出工具”
- 演进成“支持会员回馈与创作者零售的履约系统”

其主线应该是：

`Demand -> Entitlement Resolution / Routing -> Wave -> FulfillmentLine -> SupplierOrder -> Shipment -> ChannelSync / Manual Closure`

现有系统里最值得保留的资产是：

- `ProductMaster + Product` 双层商品结构
- `Wave` 作为业务批次边界
- `ReconcileWave` 的履约收敛思路

现有系统里最需要尽快停止扩张的部分是：

- 把所有流程状态塞进 `DispatchRecord.Status`
- 把所有平台语义都叫 `platform`
- 把“导出”继续当作波次终点
- 把“本系统不接手”和“系统外已完成”混成一个状态
- 把共享调整层继续扩张成第二套动态规则引擎
- 把未来 undo/redo 做成只会覆盖旧未来的线性历史栈

后续任何实施细节，只要与本计划冲突，都应优先以本计划中的业务边界、数据结构和迁移原则为准，再重新评估代码改动方向。
