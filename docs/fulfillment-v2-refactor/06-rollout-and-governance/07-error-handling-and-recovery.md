# 错误处理与恢复策略

本文件定义 V2 在异常场景下的行为约束，包括导入失败、重算失败、历史损坏和文件 I/O 失败的处理方式。

## 1. 导入失败

### 1.1 原子性原则

导入操作应作为单个 `HistoryNode` 提交。如果导入过程中发现不可恢复的格式错误，整个导入回滚，不产生 node。

### 1.2 部分成功处理

当导入数据中存在部分行格式错误时（例如 100 行中 3 行格式错误），系统应弹窗提示用户选择：

- **整体拒绝**：全部回滚，报告错误行的位置和原因，用户修正后重新导入
- **跳过错误行**：成功行正常落库并产生 `HistoryNode`，报告被跳过的行及原因

两种模式都必须支持，由用户在导入时通过弹窗做出选择。

### 1.3 错误报告内容

无论选择哪种模式，错误报告应至少包含：

- 错误行号
- 错误列（如果可定位）
- 错误原因（格式不匹配、必填字段缺失、值域超出等）
- 原始行内容摘要

## 2. ReconcileWave 重算失败

### 2.1 原子性保证

`ReconcileWave` 应在数据库事务内完成。如果中途失败：

- 事务回滚，工作区状态不变
- 不产生新的 `HistoryNode`
- 向用户报告失败原因

### 2.2 典型失败原因

- 规则引用了已不存在的商品或参与者
- 数据库写入失败（磁盘空间不足等）
- 内部逻辑断言失败（不变量被违反）

### 2.3 失败后的用户动作

- 用户可以修正导致失败的前置条件后重试
- 工作区历史不受影响（失败的重算不留痕迹）

## 3. 树状历史 patch 损坏恢复

### 3.1 恢复策略

如果某个 patch 无法应用（数据损坏、schema 不匹配等）：

1. 从最近的 checkpoint 重建到该 node 之前的状态
2. 损坏的 node 及其后续子树标记为 `corrupted`
3. 用户可选择：
   - 从其他未损坏的分支继续工作
   - 从当前数据库实际状态创建新的 root node（丢失历史但保留当前数据）

### 3.2 降级接受度

接受"从 checkpoint 重建，丢失中间细粒度历史"的降级策略。这是极端情况下的兜底方案，正常使用中不应触发。

### 3.3 预防措施

- Checkpoint 定期打（每 20 个 node）
- 被 `HistoryPin` 引用的 node 对应的 checkpoint 优先保留
- 写入 patch 后做校验（写入后立即读回验证 hash）

## 4. 文件 I/O 失败

### 4.1 基本原则

导入/导出文件操作失败时：

- 不修改工作区状态
- 不产生 `HistoryNode`
- 向用户报告具体错误

### 4.2 典型错误类型

- 文件不存在
- 权限不足
- 文件格式不匹配（期望 CSV 但内容不是）
- 文件被其他程序锁定
- 磁盘空间不足（导出时）
- 编码错误（非预期的字符编码）

### 4.3 导出失败的特殊处理

导出操作如果在文件写入阶段失败：

- 已创建的 `SupplierOrder` 对象应回滚（事务内）
- 不应出现"系统记录已导出但文件实际不存在"的状态
- 用户修正问题后可重新导出
