# 身份归并策略

本文件定义 `CustomerProfile` 跨平台身份归并的触发条件、规则和影响范围。

## 1. 业务背景

在创作者履约场景中，同一个人跨平台出现的频率不高（例如一个人既是 B 站提督又在 Gumroad 下单），尤其是数据量大起来之后占比会越来越小。但系统仍需提供归并能力，以便在确实发生时能正确处理。

## 2. 归并触发方式

### 2.1 手动归并（始终可用）

- 操作者在客户管理界面显式选择两个 `CustomerProfile` 进行合并
- 无论自动归并是否开启，手动归并始终可用
- 手动取消归并（拆分）也始终可用

### 2.2 自动归并（默认关闭，可配置）

- 默认关闭，需要用户在设置界面主动开启
- 触发条件可选，由用户配置：
  - 邮箱相同
  - 手机号相同
  - 其他跨平台可保持相对稳定和一致的标识字段
- 自动归并触发时应先进入"建议归并"状态，由用户确认后执行（而非静默合并）

## 3. 归并规则

执行归并时：

1. 保留较早创建的 `CustomerProfile` 作为主体（`primary`）
2. 被合并方的所有 `CustomerIdentity` 迁移到主体下
3. 被合并方的 `CustomerAddress` 历史迁移到主体下
4. 被合并方的 `CustomerProfile` 标记为已归并（保留记录但不再作为独立实体参与后续流程）

## 4. 对已有数据的影响

### 4.1 已有 WaveParticipantSnapshot

- 不回溯修改
- 它们是历史快照，记录的是当时的状态
- 已关闭波次中的旧快照保持原样

### 4.2 未来波次

- 归并后，该人在未来新波次中只会产生一个 `WaveParticipantSnapshot`
- 该快照关联到主体 `CustomerProfile`

### 4.3 活跃波次

- 如果归并时存在活跃波次且两个 profile 都已参与：
  - 系统应提示用户该波次中存在重复参与者
  - 由用户决定是否在当前波次中合并快照，或等波次关闭后自然生效

## 5. 可撤销性

- 归并操作本身作为一个 `HistoryNode`（scope 为全局客户管理工作区）
- 可通过 undo 撤销归并（恢复为两个独立 profile）
- 取消归并（拆分）同样作为一个 `HistoryNode`

## 6. 首版范围

首版实现：

- 手动归并和手动拆分
- 自动归并的配置界面和触发逻辑框架
- 归并建议队列（自动检测到候选后展示，用户确认后执行）

首版不实现：

- 复杂的模糊匹配算法
- 跨系统的身份关联（例如通过第三方 OAuth 自动关联）
