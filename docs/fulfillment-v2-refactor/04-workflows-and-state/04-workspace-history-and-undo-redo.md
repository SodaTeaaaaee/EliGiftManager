# 工作区历史、树状撤销与重做

本文件定义全局撤销 / 重做的行为边界、scope 规则、分支保留方式，以及终端用户可感知的交互反馈。

## 8.5 工作区历史与 Undo / Redo

### 8.5.1 目标

当前需求不是普通的线性撤销栈，而是：

- 支持 `Ctrl+Z`
- 支持 `Ctrl+Shift+Z`
- 在撤销到旧节点后重新编辑时保留旧未来分支
- 软件关闭后重新打开仍能继续使用
- 对终端用户足够轻量、默认无感
- 必要时可打开历史图显式切换分支

### 8.5.2 历史作用域应按工作区组织

全局快捷键可以统一存在，但真正撤销 / 重做的对象，应是当前激活的 `HistoryScope`。

典型例子：

- 在波次编辑里，撤销的是当前 `wave` 工作区
- 在模板配置页里，撤销的是当前 `template` 工作区
- 在商品库里，撤销的是当前 `product_library` 工作区

这样做的原因是：

- 用户对“我刚刚改了哪里”的直觉，本来就是按工作区分离的
- 不应出现“我在商品页改了东西，回波次页按 `Ctrl+Z` 却把商品页改动撤掉”的错觉

### 8.5.3 默认交互应像线性历史，但底层必须保留分支

对大多数用户来说，默认体验仍应尽量简单：

- `Ctrl+Z`
  - 回到 parent
- `Ctrl+Shift+Z`
  - 默认沿当前首选分支前进

但底层必须保留：

- 撤销到旧节点后产生的新分支
- 原未来步骤
- 切换分支的能力

也就是说：

- 默认体验看起来像线性历史
- 真正的结构则必须是树状分支历史

### 8.5.4 历史节点必须代表“用户意图”

这是实现层最重要的约束之一。

一个 `HistoryNode` 更适合表示：

- 一次批量添加规则
- 一次批量删除规则
- 一次导入
- 一次确认共享调整
- 一次地址绑定或手工闭环决策

而不应把以下内容拆成额外 node：

- 派生重算
- 状态投影刷新
- overview 统计变化
- 快照失效标记

否则会带来两个直接问题：

- 历史图难以理解
- `Ctrl+Z` 的结果和用户预期不一致

### 8.5.5 Undo / Redo 只回滚本地工作区，不回滚外部动作

当前已经确认：

- 工厂导出
- 工厂物流文件导入
- 来源渠道同步

这些外部动作的真实执行状态，本地无法通过普通 undo/redo 保证同步回滚。

因此更准确的语义应是：

- undo/redo 回滚的是本地工作区 head
- 外部对象仍然保留
- 外部对象与当前工作区如果发生偏离，应进入：
  - `basis_drift_status`
  - `review_requirement`
  这两条提示轴

换句话说：

- undo 不是“时光倒流”
- 它只是“本地工作区历史导航”

### 8.5.6 `Wave` 是首版优先 scope，但 infra 应全应用通用

当前已经确认：

- 整个软件都应逐步纳入同一套 history 基础设施
- 但优先级上应先把 `wave` scope 做稳

更稳妥的实施顺序是：

1. 先定义共享 history kernel
2. 先接入 `wave`
3. 再逐步接入 `templates / products` 等其他工作区

原因：

- `wave` 是当前业务语义最复杂、最需要分支保留的地方
- 先把最复杂场景跑通，后续其他页面更容易复用

### 8.5.7 全局快捷键不应劫持文本输入自身的撤销

这是一个必须提前定清的 UX 约束。

当焦点位于：

- `input`
- `textarea`
- `contenteditable`

时：

- `Ctrl+Z` 与 `Ctrl+Shift+Z` 应优先服务该输入控件自身
- 不应直接跳到全局工作区历史

只有当焦点不在文本编辑上下文时，才更适合触发工作区级 undo/redo。

### 8.5.8 Toast 与短期回执应成为标准反馈层

当前已经确认：

- 撤销 / 重做必须有显眼但不碍事的可感知反馈
- Toast 自动消失后，仍应在短时间内可翻出查看

更稳妥的建议结构是两层：

1. `即时 Toast`

- 每次 undo / redo 后立即出现
- 说明刚才撤销或重做了什么
- 例如：
  - `已撤销：批量为 12 件商品添加身份规则`
  - `已重做：删除 1 条共享调整`

2. `短期回执托盘`

- 不需要长期悬浮
- 但应允许用户在短时间内翻出查看最近动作
- 每条回执可进一步跳转到 history 图或再次执行 undo / redo

这里还未完全定死的，只是具体时长和条数。

当前更推荐的首版默认值是：

- Toast 停留数秒
- 回执托盘保留最近若干条动作，或保留十几分钟以内的动作

这些数值可后续再通过实际使用反馈微调。

### 8.5.9 历史图应是高级入口，而不是主流程负担

用户已经明确希望：

- 分支切换能力存在
- 但绝大多数人默认不需要天天看“版本树”

因此更合理的产品策略是：

- 默认只暴露快捷键、简洁 toast 和轻量回执
- 历史图作为高级入口按需打开
- 只有在需要切换分支、定位旧节点、查看 basis 关系时才进入

更适合的承载方式包括：

- 侧边抽屉
- 独立弹层
- 波次内高级工具页

而不建议一开始就把它做成主工作流里长期占据空间的复杂页面。

### 8.5.10 持久化是硬需求，不是可选增强

当前已经确认：

- 历史必须在软件关闭后继续保留

这意味着：

- 历史树不能只存在于前端内存
- head、分支、checkpoint、pin 状态都应持久化
- 重新打开软件后，scope 应能恢复到上次状态

### 8.5.11 历史与 basis 提示如何协同

当前更合理的关系是：

- 外部对象记录自己的 `basis_history_node_id`
- 用户 undo/redo 时，只改变当前 head
- 系统再比较：
  - 当前 head
  - basis node
  - basis projection hash

从而决定：

- 当前是否只是 `basis_drift_status = drifted`
- 还是已经升级为 `review_requirement = required`

这让两套机制能够协同：

- 树状历史负责本地导航
- basis 提示负责外部对象关联判断

### 8.5.12 首版历史系统的最小可用边界

首版更建议至少做到：

1. `wave` scope 可用
2. 树状分支不丢
3. 历史持久化
4. 全局快捷键可用
5. 不抢文本输入的原生 undo/redo
6. undo/redo 有 toast 与短期回执
7. basis 偏离能与历史节点关联
8. drift 与 review 两条提示轴能与历史节点关联

而以下能力可以延后：

- 很复杂的历史图可视化
- 全应用所有页面一次性深度接入
- 高级 merge / cherry-pick 语义
