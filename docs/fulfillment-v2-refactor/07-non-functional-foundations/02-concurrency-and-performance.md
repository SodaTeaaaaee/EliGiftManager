# 并发假设与性能约束

本文件定义 V2 的并发模型假设、数据规模预期和性能目标，用于指导实现层的技术选型与优化方向。

## 1. 并发模型

### 1.1 单写者假设

当前阶段显式声明以下约束：

- 同一时刻只有一个应用实例对同一数据库执行写入
- 桌面应用通过进程级文件锁保证独占
- 不设计分布式冲突解决（不需要 CRDT 或 OT）

Wails 本身是单进程多窗口架构，多个窗口共享同一个 Go 后端，因此"多窗口编辑同一波次"在当前技术栈下不构成并发写入问题。

### 1.2 未来多端同步的扩展空间

当前设计在架构层面为未来多端同步保留了扩展空间，但不在首版实现：

1. 树状历史天然比线性栈更适合多端场景——各端各自产生本地分支，同步时做 merge/rebase
2. `HistoryNode` 保留 `created_by` 字段，不硬编码"全局只有一个 head"的假设
3. 未来同步层是独立模块，不侵入核心历史结构

如果未来需要跨设备同步，更可能的路径是：

- 各端各自产生本地分支
- 同步时做 merge/rebase
- 而不是实时共享同一个 SQLite 文件

### 1.3 SQLite 运行时约束

- 使用 WAL 模式（已在 `07-non-functional-foundations/01-scalability-and-i18n.md` 中确认）
- 应用层设置 `busy_timeout = 5000`（5 秒）
- 备份系统应使用 SQLite Online Backup API 或 `VACUUM INTO`，避免在写入高峰期与主进程竞争

## 2. 数据规模预期

### 2.1 参与者与履约行

| 场景 | 参与者数 | 履约行数 | 说明 |
|------|----------|----------|------|
| 典型 | 50–2000 | 200–10000 | 中小型创作者日常使用 |
| 大波次 | 2000–20000 | 10000–100000 | 头部创作者常规波次 |
| 极端 | 20000–50000 | 100000–500000 | 头部创作者 + 订单型需求叠加 |

说明：

- 履约行数通常是参与者数的 4–5 倍（会员权益场景下每人多件商品）
- 订单型需求的行数可能更大（订单量通常多于会员量）
- 系统应在极端场景下仍可正常使用，但允许以进度反馈换取响应时间

### 2.2 活跃波次

- 同时存在的活跃波次：1–3 个
- 当前为单用户桌面应用，不存在多人同时编辑不同波次的需求

### 2.3 树状历史节点

| 场景 | 节点数/波次 | 说明 |
|------|-------------|------|
| 典型 | 20–60 | 一次完整波次编辑流程 |
| 复杂 | 60–200 | 反复调整、多次补充导入 |

## 3. 性能目标

### 3.1 ReconcileWave 重算

| 数据规模 | 目标延迟 | 交互策略 |
|----------|----------|----------|
| < 10000 履约行 | < 500ms | 无感，不显示加载 |
| 10000–100000 履约行 | < 5s | 显示进度指示 |
| > 100000 履约行 | < 30s | 显示进度 + 允许取消 |

实现要求：

- 批量写入而非逐行写入
- 规则 selector 预编译为 map 查找，不做逐行全扫描
- 大波次时支持进度回调

### 3.2 工作区历史

- Checkpoint 频率：每 20 个 node 打一次 checkpoint
- 单个 patch 大小：应控制在合理范围内（具体阈值在实现阶段根据实测确定）
- 从 checkpoint 恢复到任意 node 的重放链长度不超过 20 步

### 3.3 GC 策略

- 被 `HistoryPin` 引用的 node 永不 GC
- 当前 head 所在分支的所有祖先 node 永不 GC
- 未 pin 且不在当前活跃分支上的旧分支：保留至少 30 天后可压缩
- 压缩方式：合并连续 patch 为单个 checkpoint，丢弃中间细粒度 patch
- 具体阈值可在实际使用后微调
