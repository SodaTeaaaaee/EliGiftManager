# 可扩展性与多语言基础

本文件定义 V2 的两项非功能基础能力：

- 处理较大数据规模时的查询与渲染策略
- 至少支持中文与英文的 i18n 策略

这两项不是“以后有空再补”的附属优化，而是 V2 作为通用履约辅助系统必须具备的底层能力。

## 1. 为什么要把它们并入主计划

当前 V2 已经不只是“给一点点数据做复杂规则处理”。

它要同时面对：

- 会员权益型需求
- 创作者零售型需求
- 工厂提交与回传
- 来源渠道回填
- 树状工作区历史
- profile / template / service 分层

如果这些能力只在小数据量下成立，那么系统的通用性会被高估。

因此，V2 必须同步具备：

1. 大数据可扩展性

- 列表查询不能默认依赖全量前端持有
- 排序、过滤、分页必须支持后端事实来源
- 远程滚动、游标分页、轻量 DTO 都应纳入长期设计

2. 多语言可扩展性

- UI 文案、提示、错误、空状态、导出模板文字应支持本地化
- 至少应支持 `zh-CN` 与 `en-US`
- 业务代码、状态码、数据库键名不应直接使用可变的展示语言文本

## 2. 大数据能力应该怎么并入 V2

### 2.1 不是全局强制，而是按工作区和页面分层接入

不是每个页面都必须立刻切成远程滚动。

更合理的方式是：

- 小数据页面保留轻量本地方案
- 大结果集页面切到远程查询方案
- 同一套查询状态模型和数据源抽象可在两种模式间切换

### 2.2 必须优先支持的页面类型

未来最容易吃到大数据压力的页面，包括但不限于：

- 会员 / 买家列表
- 波次内聚合总览
- 波次内调整与复核页
- 导出 / 回填 / 物流状态队列
- 历史图与 basis 关联视图

这些页面应该优先具备：

- 后端过滤
- 后端排序
- 后端分页或游标分页
- 轻量列表 DTO
- 按需加载详情

### 2.3 大数据能力的核心原则

1. 查询真相由后端决定

- 前端不应再对全量数据做最终排序和最终分页

2. DTO 必须轻量

- 列表页只返回当前视图需要的最小字段
- 地址、昵称历史、复杂嵌套详情应按需加载

3. 页面模式可以不同，但查询模型要统一

- 同一套 `TableQueryState` 可驱动本地与远程模式
- 不同页面只是在数据源层切换实现

4. 不能为了大数据牺牲业务语义

- 大数据模式的目标是让语义继续可用
- 不是把业务模型压扁成只剩分页

## 3. i18n 为什么是好主意

是。

但前提是：i18n 应该是**展示层和文案层的能力**，不是把业务语义本身国际化成不稳定文本。

### 3.1 应该国际化的内容

- 页面标题
- 按钮文案
- 表头
- 状态标签
- toast
- 空状态
- 校验提示
- 帮助文案
- 导出模板中给用户看的字段名

### 3.2 不应该国际化成可变文本的内容

- 数据库主键
- 枚举 code
- 路由判断 code
- 业务规则判断键
- profile key
- source channel / demand kind 等稳定语义键

这些内容应保持语言中立，只在展示时翻译。

### 3.3 推荐的最小语言范围

- `zh-CN`
- `en-US`

先把这两种语言的 UI、提示与模板输出打通，比一开始做得很宽更重要。

## 4. 对 profile / template 层的影响

多语言不应该让 profile 变成“翻译字典”。

更合理的做法是：

- `IntegrationProfile` 负责声明来源业务面和能力边界
- `DocumentTemplate` 负责某种语言或某种输出格式的具体字段映射
- UI 的翻译资源由前端 i18n 系统管理

如果某个业务面存在中英两套模板或标题差异，可以在模板层做 locale variant，但底层业务 code 仍然保持稳定。

[`docs/BACKEND-TABLE-ARCHITECTURE-TODO.md`](../../BACKEND-TABLE-ARCHITECTURE-TODO.md) 仍然保留，但它现在应该被视为：

- 而不是与重构计划分离的孤立备忘录

也就是说：

- 该文档负责更细的表格查询与分页技术路线
- 本文档负责把这条路线放回 V2 的整体基础能力里

## 6. 落地原则

1. 业务语义先稳定，语言再本地化
2. 查询状态先统一，页面模式再分化
3. 代码与枚举保持语言中立
4. 展示文本通过 i18n 资源层解析
5. 大数据方案与小数据方案可以并存，但底层抽象必须共享
