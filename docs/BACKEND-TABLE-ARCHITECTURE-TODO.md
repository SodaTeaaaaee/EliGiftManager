## 1. 文档目的

本文档记录项目未来在“大数据场景”下的表格架构改造方向。目标方案为：

- 后端统一负责：
  - 过滤
  - 排序
  - 分页
  - 惰性加载 / 分块加载
- 前端统一负责：
  - 展示
  - 查询状态管理
  - 受控排序 UI
  - 远程数据请求
  - 滚动容器 / 固定行高分页的视图实现

多语言本身不属于本文件的核心主题，但它会影响：

- 表头文案
- 空状态
- 错误提示
- 分页文案
- 导出模板的展示文本

因此，表格架构的技术方案需要与 V2 的 i18n 方案协同，而不是把显示语言硬编码进查询层。

本文档是未来改造的设计草案、执行 TODO、约束说明和决策备忘录。
现阶段项目仍可优先使用纯前端方案，但一旦数据规模、页面复杂度、离屏测量成本继续增长，本方案应成为正式演进方向。

## 1.1 作用域限制

本文档只讨论以下问题：

- 表格查询职责归前端还是后端
- 远程排序、分页、滚动如何设计
- 大数据列表 DTO 如何收敛
- 统一查询状态模型如何建立

本文档不负责定义以下内容：

- 会员、商品、波次、发货记录的业务语义
- 模板系统的业务边界
- 当前分支是否应立刻重构某个领域模型
- 发货规则本身应该如何设计

## 2. 为什么需要这份方案

当前项目已经存在两类不同需求：

1. 小中型数据集

- 适合前端一次性拿全量数据
- 适合前端离屏测量
- 适合“精确自适应分页”

2. 大数据集

- 不适合前端一次性拿全量数据
- 不适合全量离屏测量
- 不适合让前端承担排序、分页的最终职责

当前纯前端自适应分页方案的核心特征是：

- 客户端拥有当前结果集全量数据
- 客户端在当前宽度下离屏渲染所有行
- 客户端测量每一行真实高度
- 客户端根据全量真实行高重新打包分页

这一方案在中小数据量下是合理的，但在大数据场景下有天然上限：

- 网络成本高：需要一次性传输全量数据
- 内存成本高：前端长期持有全量行对象
- 测量成本高：需要渲染并测量全量行
- 首屏成本高：必须等大量数据到齐才能得到最终分页
- 职责边界模糊：排序和分页最终由前端承担，不利于前后端职责清晰分离

因此，未来必须准备一条纯后端数据查询架构路线。

---

## 3. 本方案的核心结论

### 3.1 目标架构

未来大数据模式下，项目应改造成：

- 后端是表格查询的唯一事实来源
- 前端不再对全量数据做最终排序和最终分页
- 前端不再依赖全量离屏测量来生成精确分页

### 3.2 必须接受的取舍

如果选择纯后端排序分页方案，则必须接受：

- 放弃“精确自适应分页”
- 放弃“基于真实换行高度的全量分页打包”
- 改用：
  - 远程滚动 / 惰性加载
  - 或固定行高 + 估算每页条数的近似分页

也就是说，本方案的根本取舍是：

- 保留大数据可扩展性
- 放弃当前高精度自适应分页模型

这是架构层面的主动权衡，不是实现缺陷。

---

## 4. 本方案的适用时机

满足以下任一条件时，应认真考虑从纯前端方案切向本方案：

1. 单表结果集常态达到数千到数万行
2. 单行内容复杂，离屏测量明显拖慢首屏
3. 页面存在重对象 payload，前端内存和序列化成本明显上升
4. 客户端排序和分页开始与后端查询语义冲突
5. 未来明确需要：
   - 游标分页
   - 无限滚动
   - 虚拟列表
   - 大数据统一查询接口

### 4.1 当前项目中最可能优先切换的页面

按当前业务形态判断：

- `MemberCrmPage` 是最可能最早需要切换的页面
  - 当前列表 payload 较重
  - 后续会员总量增长概率高
  - 排序、过滤、详情查询天然更适合前后端职责分离

- `WaveExportStep` 是否切换取决于单波次记录量
  - 如果单波次记录量仍是中小规模，则未必需要切换
  - 如果未来出现超大波次导出，则也可逐步切换

---

## 5. 非目标

本方案明确不追求以下目标：

1. 不追求继续保留当前精确自适应分页
2. 不追求前端在大数据模式下继续拥有全量真实行高
3. 不追求所有页面永远使用一模一样的分页实现
4. 不追求把当前所有前端离屏测量逻辑直接迁移到后端

本方案的重点是：

- 清晰的前后端职责边界
- 大数据可扩展性
- 统一的查询状态入口
- 可控的远程排序 / 分页 / 滚动逻辑

---

## 6. 新旧架构对比

## 6.1 当前纯前端方案

数据流：

`后端返回全量 -> 前端过滤 -> 前端排序 -> 前端离屏测量 -> 前端分页 -> 渲染`

优点：

- 可以做精确自适应分页
- 小中型数据体验很好
- 排版精致

缺点：

- 不适合大数据
- 前端职责偏重
- 首屏依赖全量数据和全量测量

## 6.2 未来纯后端方案

数据流：

`前端提交查询状态 -> 后端过滤/排序/分页 -> 前端只渲染当前页/当前块`

优点：

- 职责清晰
- 更符合前后端分离
- 易于扩展到大数据
- 更适合游标分页和无限滚动

缺点：

- 不能再做当前那种精确自适应分页
- 需要引入近似分页或远程滚动
- 需要改造现有表格模式语义

---

## 7. 核心设计原则

未来改造必须遵循以下原则：

1. 查询真相只能有一个来源

- 最终显示顺序由后端决定
- 最终分页边界由后端决定

2. 排序必须先于分页

- 前端不能再对已分页结果做“伪排序”

3. 所有远程表格必须共享统一查询状态模型

- 不能每个页面发明自己的排序、翻页、过滤协议

4. 大数据模式下，前端只保存必要数据

- 列表页尽量使用轻量 DTO
- 详情信息按需加载

5. UI 交互可以统一，但底层数据源策略不必强行相同

- 小数据页面仍可继续使用前端模式
- 大数据页面可切到纯后端模式

6. 必须从现在开始预留“本地数据源 / 远程数据源”切换能力

- 避免未来再推翻一次整套表格状态模型

---

## 8. 未来表格模式重新定义

当前设置中的表格模式为：

- `scroll`
- `paginated`

在未来纯后端大数据模式下，这两个模式的语义需要重定义。

## 8.1 `scroll`

未来语义：

- 远程滚动模式
- 支持：
  - 后端过滤
  - 后端排序
  - 后端分页 / 游标分页
  - 惰性加载
  - 虚拟列表（推荐）

适用：

- 大数据
- 高并发查询
- 超长结果集

## 8.2 `paginated`

未来语义：

- 远程近似分页模式
- 使用固定行高或受控行高估算每页条数
- 不基于真实离屏测量打包分页

适用：

- 用户仍然偏好“页”的概念
- 但数据量已不适合全量前端测量

## 8.3 明确放弃的能力

大数据模式下，`paginated` 不再意味着：

- 根据真实内容高度精确分页
- 根据当前宽度重新打包整份数据
- 页边界与真实自然换行完全一致

必须明确告知未来维护者：

> 纯后端模式下的 `paginated` 是近似分页，不是精确自适应分页。

---

## 9. 统一查询状态模型

未来所有表格，无论前端数据源还是后端数据源，都应共享同一套查询状态模型。

建议定义：

```ts
export type TableMode = "scroll" | "paginated";

export type SortOrder = "ascend" | "descend";

export interface UserSortClause {
  key: string;
  order: SortOrder;
}

export interface TableQueryState {
  keyword?: string;
  filters: Record<string, unknown>;
  userSorts: UserSortClause[];
  page: number;
  pageSize: number;
  mode: TableMode;
}
```

说明：

- 第一阶段 UI 可以只允许 `userSorts` 长度为 0 或 1
- 但底层结构直接设计成数组，方便未来升级多列复合排序

---

## 10. 统一排序状态模型

排序系统必须从“受控 UI 状态”开始设计，而不是依赖 UI 组件内部黑箱。

建议统一：

```ts
export interface TableSortState {
  userSorts: UserSortClause[];
}
```

### 10.1 第一阶段的 UI 能力

- 用户可见能力：单列三态排序
- 内部数据结构：可扩展到多列复合

### 10.2 单列三态排序的语义

表头点击循环：

- `未排序`
- `升序`
- `降序`
- `未排序`

### 10.3 未来多列复合排序的语义

如果未来升级，建议语义如下：

- 普通点击：替换主排序
- `Shift + 点击`：追加次级排序

例如：

- `平台 asc`
- `地址状态 desc`
- `更新时间 desc`

### 10.4 稳定尾排序键

即使用户只看到单列排序，后端也应自动追加稳定尾键，避免相同主键值下顺序随机变化。

推荐：

- `updated_at DESC`
- `id DESC`

示例：

用户请求：

```json
[{ "key": "latestNickname", "order": "ascend" }]
```

后端实际执行：

```sql
ORDER BY latest_nickname ASC, updated_at DESC, id DESC
```

这样即使主排序值相同，结果顺序也可预测。

## 11. 远程数据源抽象

未来前端必须抽象统一数据源接口，而不是让页面直接依赖某个后端函数。

建议：

```ts
export interface RemoteTablePage<T> {
  items: T[];
  total: number;
  page: number;
  pageSize: number;
  hasMore?: boolean;
  nextCursor?: string | null;
}

export interface RemoteTableDataSource<T> {
  load(query: TableQueryState): Promise<RemoteTablePage<T>>;
}
```

如果未来支持游标模式，也可再扩展：

```ts
export interface CursorTableQueryState {
  keyword?: string;
  filters: Record<string, unknown>;
  userSorts: UserSortClause[];
  cursor?: string | null;
  limit: number;
  mode: "scroll";
}
```

## 12. 后端 API 设计原则

未来所有大数据表格接口都应遵循一致规范。

## 12.1 请求参数

建议请求结构至少包含：

- `keyword`
- `filters`
- `sort`
- `page`
- `pageSize`

或在滚动模式下：

- `keyword`
- `filters`
- `sort`
- `cursor`
- `limit`

建议统一成显式结构，而不是继续使用多个松散 positional 参数。

### 12.1.1 推荐分页请求 DTO

```go
type SortClause struct {
	Key   string `json:"key"`
	Order string `json:"order"` // ascend / descend
}

type TableQuery struct {
	Keyword  string                 `json:"keyword"`
	Filters  map[string]any         `json:"filters"`
	Sorts    []SortClause           `json:"sorts"`
	Page     int                    `json:"page"`
	PageSize int                    `json:"pageSize"`
}
```

### 12.1.2 推荐分页响应 DTO

```go
type TablePage[T any] struct {
	Items    []T   `json:"items"`
	Total    int64 `json:"total"`
	Page     int   `json:"page"`
	PageSize int   `json:"pageSize"`
}
```

### 12.1.3 推荐游标响应 DTO

```go
type CursorPage[T any] struct {
	Items      []T    `json:"items"`
	HasMore    bool   `json:"hasMore"`
	NextCursor string `json:"nextCursor"`
}
```

## 13. 后端职责边界

未来大数据模式下，后端必须负责：

1. 对外暴露统一查询接口
2. 验证排序 key 合法性
3. 将前端排序模型映射为 SQL 排序
4. 自动追加稳定尾排序键
5. 对查询结果做最终分页
6. 返回轻量列表 DTO

后端不负责：

1. 前端表头 UI 状态
2. 当前窗口宽度
3. 真实换行后的行高
4. 前端滚动容器布局

---

## 14. 前端职责边界

未来大数据模式下，前端必须负责：

1. 维护受控查询状态
2. 表头排序交互
3. 将排序状态转为后端 query model
4. 渲染当前页或当前块结果
5. 渲染近似分页控件或远程滚动容器
6. 在 `scroll / paginated` 模式间切换不同远程视图策略

前端不再负责：

1. 对全量结果做最终排序
2. 对全量结果做最终分页
3. 对全量结果做离屏真实高度测量

## 15. 固定行高 + 近似分页设计

这是纯后端方案下最重要的分页替代策略。

## 15.1 基本思路

不再测量真实内容高度，而是为某张表定义一个稳定的“目标行高”：

- 例如 52px
- 或 small / medium 两种固定行高

然后根据表格视口高度估算：

```text
estimatedPageSize = floor(availableBodyHeight / rowHeight)
```

再把这个 `estimatedPageSize` 作为查询参数发给后端。

## 15.2 这一方案的本质

这是“视觉上自适应、语义上近似分页”的方案：

- 页面高度变化时，每页条数会变化
- 但不是根据真实换行精确计算
- 只是根据固定行高估算

## 15.3 必须满足的前提

为了让固定行高方案可控，表格必须尽量限制单行高度波动：

1. 文字单元格尽量单行或两行截断
2. 不允许任意高度自由增长
3. 多 tag、多图片、多段文本的场景不适合走该模式

## 15.4 适合固定行高分页的页面

- `MemberCrmPage`
- 简单列表型 `WaveExportStep`
- 未来简单运营列表页

## 15.5 不适合固定行高分页的页面

- `WaveTagStep`
- 强依赖多行换行和 tag 堆叠的复杂表格

这些页面在未来大数据模式下更适合：

- 滚动模式
- 虚拟滚动

而不是近似分页。

---

## 16. 远程滚动 / 虚拟滚动设计

对于真正的大数据页面，推荐优先使用远程滚动，而不是近似分页。

## 16.1 推荐形态

- 后端返回当前块
- 前端维护块缓存
- 视口只渲染可见行
- 滚动到底部时请求下一块

## 16.2 为什么推荐虚拟滚动

它能同时降低：

- 首屏渲染成本
- DOM 数量
- 内存压力
- 页面滚动卡顿

## 16.3 与近似分页的关系

未来可允许：

- `scroll` 模式：远程虚拟滚动
- `paginated` 模式：远程近似分页

这样设置里的模式开关仍然有意义，但两种模式都不再使用当前精确自适应分页。

---

## 17. 列表 DTO 轻量化要求

未来切大数据模式前，必须先对列表页 payload 做轻量化。

### 17.1 为什么必须轻量化

如果列表接口继续直接返回：

- 地址历史
- 昵称历史
- 大对象嵌套

那无论前端排序还是后端分页，整体成本都会偏高。

### 17.2 `MemberCrmPage` 的改造要求

当前 `MemberCrmPage` 列表页的 `MemberItem` 包含：

- `Addresses`
- `Nicknames`

这在大数据模式下不合适。

未来建议拆成：

#### 列表 DTO

只保留：

- `id`
- `platform`
- `platformUid`
- `latestNickname`
- `activeAddressCount`
- `latestRecipient`
- `latestPhone`
- `latestAddress`
- `updatedAt`

#### 详情 DTO

抽屉打开时按需加载：

- 地址列表
- 昵称历史
- 其他详情字段

这样可以显著降低：

- 列表首屏 payload
- 前端内存压力
- 序列化开销

## 18. 排序字段白名单与映射

后端不能接受前端传任意字符串直拼 SQL。

必须维护每个列表接口的排序字段白名单。

例如 `MemberCrmPage`：

```go
var memberSortKeyMap = map[string]string{
	"latestNickname":     "members.latest_nickname",
	"platform":           "members.platform",
	"platformUid":        "members.platform_uid",
	"activeAddressCount": "active_address_count",
	"latestRecipient":    "latest_recipient",
	"latestPhone":        "latest_phone",
	"latestAddress":      "latest_address",
	"updatedAt":          "members.updated_at",
}
```

如果字段本身不是表内天然列，而是派生值：

- 要么后端提前计算
- 要么查询时做子查询 / JOIN
- 要么重构表结构引入冗余字段

## 19. `MemberCrmPage` 的纯后端方案

`MemberCrmPage` 是最适合作为纯后端方案试点的页面。

## 19.1 目标

- 查询、过滤、排序、分页全部后端完成
- 前端只保留：
  - 搜索框
  - 平台筛选
  - 缺地址开关
  - 排序表头 UI
  - 分页 / 滚动视图

## 19.2 推荐改造顺序

1. 新增轻量列表接口
2. 新增详情接口
3. 新增统一排序查询参数
4. 前端改为受控远程查询
5. 再根据数据规模选择：
   - 远程近似分页
   - 远程滚动

## 19.3 远程近似分页模式

适合：

- 数据量较大，但用户仍依赖“页”的概念

实现：

- 视口高度 -> 估算 pageSize
- 前端请求指定页
- 后端返回该页结果和总数

## 19.4 远程滚动模式

适合：

- 数据量极大
- 页码意义弱
- 更强调浏览速度

实现：

- 游标或 offset/limit
- 虚拟滚动容器
- 分块缓存

## 20. `WaveExportStep` 的纯后端方案

`WaveExportStep` 是否需要切纯后端方案取决于单波次数据量。

## 20.1 如果波次记录量仍较小

不建议优先切。

理由：

- 业务复杂度不高
- 当前全量前端仍可接受
- 精确自适应分页体验更好

## 20.2 如果波次记录量未来很大

可按以下方案改造：

- 后端提供：
  - 关键字过滤
  - 平台过滤
  - 地址状态过滤
  - 远程排序
  - 远程分页

- 前端改用：
  - 远程滚动
  - 或近似分页

但注意：

- `WaveExportStep` 的导出模板选择区、同步地址按钮、状态摘要区，仍可保留在表格上方
- 只是列表内容本身不再走全量离屏测量

## 21. 未来统一基础设施建议

为避免未来重复改造，应提前抽以下基础设施。

## 21.1 `useTableQueryState`

职责：

- 管理：
  - keyword
  - filters
  - sort state
  - mode
  - page / pageSize

## 21.2 `useRemoteTableData`

职责：

- 接收 `TableQueryState`
- 调用远程数据源
- 维护：
  - loading
  - current items
  - total
  - hasMore
  - error

## 21.3 `useTableSort`

职责：

- 统一单列三态切换
- 未来可扩展多列复合排序
- 输出受控 `sortOrder` 给列定义

## 21.4 `buildRemoteSortableColumns`

职责：

- 将统一排序状态映射到 Naive UI 列定义
- 给列注入：
  - `sortOrder`
  - `sorter`
  - 交互回调

这样未来无论本地表格还是远程表格，都能复用同一套排序入口。

## 23. 关键风险

## 23.1 用户认知风险

风险：

- 用户会把未来的 `paginated` 理解成当前精确自适应分页

应对：

- 明确文案和说明
- 必要时区分：
  - `精确自适应分页`
  - `固定行高分页`

## 23.2 复杂表格不适配风险

风险：

- 像 `WaveTagStep` 这种高度高度依赖内容的表格，不适合固定行高分页

应对：

- 对复杂表格优先使用远程滚动
- 不强行让所有表格都支持两种模式

## 23.3 后端排序字段实现复杂度

风险：

- 某些派生列不容易直接 SQL 排序

应对：

- 白名单映射
- 轻量 DTO
- 需要时使用冗余字段或物化派生字段

## 23.4 前端与后端状态不同步风险

风险：

- UI 排序状态和后端真实查询状态脱节

应对：

- 统一受控 query model
- 所有远程请求都由同一状态驱动

## 24. 验收标准（未来切方案 C 时）

如果未来正式落地纯后端方案，应至少满足以下验收条件：

1. 前端不再对当前结果集做最终排序
2. 后端排序结果稳定，主排序值相同下不会随机抖动
3. 前端分页或滚动仅消费后端返回结果
4. 列表页 payload 明显轻量化
5. `MemberCrmPage` 详情信息按需加载
6. 大数据场景下首屏时间、滚动性能优于旧前端全量测量方案
7. 设置中的表格模式语义清晰，不误导用户

---

## 25. 当前结论

截至本文档撰写时，项目的现实判断如下：

1. 当前仍可优先使用纯前端方案

- 尤其是 `WaveExportStep`
- 以及中小规模数据下的 `MemberCrmPage`

2. 未来纯后端方案是合理且必要的长期演进方向

- 更符合前后端职责分离
- 更适合大数据
- 更适合统一远程查询协议

3. 一旦转入纯后端方案，必须接受：

- 放弃当前精确自适应分页
- 采用固定行高近似分页或远程滚动

4. 未来真正的架构关键，不是“要不要后端排序”本身，而是：

- 是否建立统一查询状态模型
- 是否抽象统一数据源接口
- 是否先把列表 DTO 轻量化
