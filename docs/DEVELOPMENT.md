# Development Workflow

This repository uses Go for the backend, Wails for the desktop shell, and a Vue 3 frontend built by Vite under Deno. The codebase is already structured for a real desktop application, but the product logic is still under active exploration.

## Product Caveat

- Business logic is not final yet. Database models and service helpers exist, but they should be treated as a working draft rather than a locked domain model.
- The current frontend UI is a temporary shell. Expect major layout, copy, navigation, and interaction changes while the product direction is still being refined.

## Setup

From the repository root:

```powershell
go mod tidy
Set-Location frontend
deno install
Set-Location ..
```

## Daily Commands

- `wails dev`: starts the desktop app and proxies the frontend development server
- `cd frontend && deno task dev`: runs only the Vite dev server on `127.0.0.1:5173`
- `cd frontend && deno task typecheck`: runs `vue-tsc`
- `cd frontend && deno task build`: builds the frontend for production
- `go test ./...`: runs backend tests
- `wails build`: builds the packaged desktop binary

## Generated Artifacts

- `frontend/node_modules`: generated by Deno's npm compatibility layer
- `frontend/package.json.md5`: generated by Deno bookkeeping
- `frontend/dist`: generated by Vite production builds
- `frontend/wailsjs`: generated by Wails bindings
- `.cache`: optional local build and test caches
- `build/bin`: generated packaged binaries

Only `frontend/wailsjs` is committed, because the frontend imports the generated bridge files. The other generated outputs stay ignored.

## Frontend Boundaries

- `src/app`: app bootstrap, layout, and router
- `src/pages`: route-level pages
- `src/shared`: reusable UI, types, and infrastructure helpers
- `src/shared/lib/wails/app.ts`: centralized Wails application wrapper

Keep direct imports from `wailsjs` inside `src/shared/lib/wails`. Route pages and composables should call the wrapper layer, not generated bindings directly.

## Working Rules

- Prefer using local `.cache` directories when running expensive build or test commands in development automation.
- Treat current route content as mock or placeholder UI unless the page is explicitly wired to backend data.
- When updating desktop-facing flows, document whether the change affects only scaffolding or actual product logic.
