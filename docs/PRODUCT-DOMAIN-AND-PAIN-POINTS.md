# 软件目的、业务域与核心痛点

> 本文档是当前分支最重要的业务域说明。
> 如果未来遇到旧注释、TODO、临时实现、模板实验代码或历史讨论与本文冲突，
> 以本文、[`internal/model/tables.go`](../internal/model/tables.go) 和当前 service / controller 的实际行为为准。

## 1. 这款软件到底在解决什么问题

EliGiftManager 不是一个通用电商后台，也不是直播间“礼物”管理工具。
它解决的是内容创作者的一个非常具体、非常重复、而且极易出错的线下履约问题：

- 创作者每个月都会从多个会员平台导出高等级支持者名单。
- 这些支持者会按照不同赞助档位，获得不同数量或不同组合的实体周边。
- 周边并不是在系统里直接生产出来的，而是来自不同代发工厂或商品平台。
- 每个工厂要求的导出表头、字段顺序、文件格式都可能不同。
- 创作者或运营需要在“会员名单”和“工厂商品”之间建立可追踪、可复算、可导出的对应关系。

这类工作如果靠手工 Excel 处理，会长期面临以下问题：

- 多平台列名、格式、昵称习惯完全不统一。
- 会员等级会变化，不能把“本月身份”误认为“永久身份”。
- 同一个用户可能多次改昵称、多次改地址，手工对照非常痛苦。
- 同一款商品会在多个波次反复出现，如果没有全局商品库，就会持续重复录入和重复传图。
- 不同工厂只认自己的 SKU 和导出模板，平台内部的名字并不足以直接发货。
- 一旦中途改了分配规则，很难知道哪些记录需要重新导出、哪些只是待补地址。

软件的目标不是“替代一切表格”，而是把这些跨平台、跨波次、跨工厂的重复劳动压缩成一套稳定的数据流：

`导入原始数据 -> 归一化 -> 波次内分配 -> 生成发货明细 -> 按工厂模板导出`

## 2. 目标用户与典型使用场景

主要用户不是传统电商商家，而是：

- VTuber
- B 站 UP 主
- YouTube 创作者
- 小型工作室或运营助理
- 需要长期按月给赞助会员寄送实体回馈的个人或团队

典型场景通常具有以下特征：

- 每月或每个活动周期都会有一次新的发货任务。
- 同一个会员会持续存在，但本月等级、本月昵称、本月地址是否有效都可能变化。
- 同一款周边会在多个波次重复使用，但图片、命名、导入来源可能并不总是完全干净。
- 最终导出不是给会员看的，而是给工厂或代发平台直接使用的打单文件。

## 3. 一个完整波次的业务流程

当前项目的核心业务可以按下面的顺序理解。

### 3.1 创建波次

`Wave` 表示一次具体的发货任务，例如：

- `2026 年 5 月舰长回馈`
- `周年纪念提督补发`
- `活动限定波次`

波次不是抽象概念，它是实际的履约边界：

- 哪些会员参与本次发货
- 本次用哪些商品
- 本次按什么等级规则分配
- 本次最终导出了哪些发货明细

都会被绑定在一个具体 `Wave` 下。

### 3.2 导入会员名单

会员来源可能是 Bilibili、大航海、YouTube Membership 或其他平台导出的表格。

导入后会发生两层落库：

- 全局层：
  - 写入或复用 `Member`
  - 通过 `(platform, platform_uid)` 唯一识别同一个粉丝
  - 昵称历史进入 `MemberNickname`
  - 地址历史进入 `MemberAddress`
  - 非核心杂项字段进入 `ExtraData`
- 波次层：
  - 为本次波次生成 `WaveMember`
  - 把本次导入时看到的 `GiftLevel`、昵称、平台 UID 等快照落在这里

这一步最重要的原则是：

- `Member` 是“这个人是谁”
- `WaveMember` 是“这个人在这次波次里以什么身份参与”

### 3.3 导入商品与商品图片

商品来源是工厂或商品平台导出的数据。
当前分支已经采用“双层商品模型”：

- `ProductMaster`
  - 全局商品库
  - 用 `(platform, factory_sku)` 做稳定去重
  - 代表跨波次复用的商品主档
- `Product`
  - 波次内商品快照
  - 用 `(wave_id, platform, factory_sku)` 去重
  - 代表本次发货任务实际使用的商品版本

这样做的原因很重要：

- 商品天然具有跨波次复用价值，应该有全局主档。
- 但波次导入仍然是业务快照，不能让一次临时导入、临时命名或临时图片污染所有历史波次。
- 波次删除时，不应该把全局商品库一起删掉。

当前实现的成熟边界可以理解为：

- `ProductMaster` 负责“这款商品在系统里的长期身份”
- `Product` 负责“这款商品在某次波次里的实际参与状态”

### 3.4 导入图片并建立内部资源关联

图片是业务里最容易引发误解的一部分。

当前分支的约束是：

- 只有 `柔造` 平台的商品图片导入，才会在导入阶段使用“文件名匹配商品名”的逻辑。
- 这是因为该数据源在导入时没有更可靠的额外信息可用于匹配。
- 匹配成功后，图片会去重存入内部资源目录，并分别关联到：
  - 波次快照级 `ProductImage`
  - 全局商品级 `ProductMasterImage`（当 `ProductMasterID` 存在时）

因此，文件名匹配只是导入期的临时桥梁，不是长期主键。

导入完成后，后续逻辑应该依赖的是：

- 内部图片路径
- `Product` / `ProductMaster` 的数据库关联
- 系统生成的内部 ID

而不是继续依赖原始 ZIP 里的文件名。

### 3.5 配置分配规则

会员拿到什么、拿多少，不是写死在导入器里的，而是由 `ProductTag` 驱动。

`ProductTag` 有两类核心用途：

- `level`
  - 面向赞助档位的规则
  - 例如“提督自动分配 2 个立牌”
- `user`
  - 面向某个具体 `WaveMember` 的单独修正
  - 例如补发、减发、例外覆盖

这一步是“业务策略层”，不是“导入映射层”。
也就是说，模板系统负责把原始外部列映射到内部字段，
而真正决定某个用户最终拿到什么的是波次内的分配规则和对账结果。

### 3.6 对账并生成发货真相

所有规则最终都应该汇总到 `DispatchRecord`。

`DispatchRecord` 是当前系统的单一事实来源（SSOT）。
如果要回答下面这些问题，最终都应该查它：

- 这个波次最终要给谁发货？
- 要发哪款商品？
- 发多少件？
- 用的是哪个地址？
- 这条记录是否已经导出过？

不要把 `WaveMember`、`ProductTag`、导入表格、前端临时列表当作最终发货真相。
它们都只是生成或辅助 `DispatchRecord` 的中间层。

### 3.7 绑定地址与补全状态

地址问题不是会员导入的一次性动作，而是发货履约前的关键校验步骤。

一个 `DispatchRecord` 可能处于以下状态：

- `pending_address`
  - 还没有可用地址，或者绑定的地址已失效
- `pending`
  - 地址已就绪，待发货 / 待导出
- `exported`
  - 已经按某个工厂模板导出

当前分支的 `Wave.Status` 不是独立录入的业务字段，而是由发货记录集合推导出的进度视图：

- 没有 `DispatchRecord` 时为 `draft`
- 只要存在缺地址记录，则波次为 `pending_address`
- 全部记录都已导出时，波次为 `exported`
- 其他情况为 `pending`

这意味着：

- 波次状态是汇总结论
- `DispatchRecord.Status` 才是更细粒度的底层真相

### 3.8 按平台分别导出工厂文件

导出不是“一个总表发天下”，而是按商品平台 / 工厂平台分别导出。

这也是为什么最终导出行虽然会汇总会员字段、商品字段、地址字段，
但通常不会保留“会员平台列”或“商品平台列”作为导出目标字段：

- 每个导出模板本来就只对应一个具体平台
- 同一波次可能要分别导出多个平台文件
- 导出时已经隐含了“当前文件属于哪个平台”

当前分支还遵循一个重要规则：

- 某个平台模板导出成功后，只会把该平台对应商品的 `DispatchRecord` 标记为 `exported`
- 只有当整个波次下所有平台对应的记录都完成导出，`Wave.Status` 才会变成 `exported`

## 4. 软件真正要消除的核心痛点

这套系统的价值不在于“可以导入 CSV”，而在于它系统性地削弱了几类长期痛点。

### 4.1 平台之间字段叫法完全不同

不同平台导出的会员名单、商品表、工厂表头可能完全不一致。
如果每次都靠临时写脚本或手工改 Excel，长期维护成本极高。

### 4.2 同一个粉丝在不同波次中的身份会变化

昵称会变、等级会变、地址会变。
如果没有“全局实体 + 波次快照”的分层，很容易把历史事实和当前事实混在一起。

### 4.3 同一款商品会反复出现

没有全局商品库时，团队会持续重复导入、重复上传图、重复修字段。
但如果只有全局商品库、没有波次快照，又会让某次导入的脏数据污染所有历史记录。

### 4.4 发货明细需要有唯一真相

运营经常会问：

- “这个用户到底这次有没有发？”
- “为什么数量变了？”
- “这个波次到底还有多少没导？”

如果没有 `DispatchRecord` 这样的收敛点，这些问题在中间规则和临时表格之间很难回答。

### 4.5 地址补全和导出进度是两个不同维度

一个用户可能已经完成分配，但还没填地址。
也可能地址齐了，但某个平台的工厂单还没导出。
系统必须把这些状态分开，而不是只给一个模糊的“完成 / 未完成”。

## 5. 术语与常见误解

下面这些词未来最容易被误解，必须统一。

### 5.1 `Gift` / `Product`

这里指的是实体周边，例如：

- 吧唧
- 立牌
- 挂件
- 明信片

它不是直播间的虚拟礼物，不是打赏流水，不是舰长礼物特效。

### 5.2 `GiftLevel`

这里指赞助档位 / 会员阶级，例如：

- 舰长
- 提督
- 总督
- Tier 1 / Tier 2 / Tier 3

它的业务意义是“决定这次波次如何分配商品”，
不是一个应该永久挂在全局 `Member` 上的稳定身份字段。

### 5.3 `Wave`

`Wave` 是一次具体发货任务，不是抽象活动标签，也不是商品分组。
它是这次履约的业务边界。

### 5.4 `Platform`

当前项目里至少有两类不同语义的“平台”：

- 会员平台：
  - 例如 Bilibili、YouTube
  - 用于标识 `Member` / `WaveMember` 的来源
- 商品 / 导出平台：
  - 例如 柔造
  - 用于标识 `ProductMaster` / `Product` 的来源与导出模板归属

这两类平台不是同一个维度，不要混用。

### 5.5 `FactorySKU`

这是当前商品长期身份最可靠的外部唯一键之一。
当前实现把 `(platform, factory_sku)` 作为全局商品主档的去重基础，
比“商品名称”可靠得多。

### 5.6 `TemplateConfig`

模板系统不是本软件的核心业务本体，而是一个“映射层”。
它负责把各种外部文件格式翻译成系统内部字段，
但它不决定：

- 会员是谁
- 某波次等级是什么
- 商品如何分配
- 最终应该发什么

这些问题依然属于核心领域模型和业务规则。

## 6. 当前分支的权威数据模型

下面这组理解方式可以直接指导后续维护。

### 6.1 `Member`

回答的问题：

- “这个粉丝在全局范围内是谁？”

关键特征：

- 通过 `(platform, platform_uid)` 唯一识别
- 不持久化波次级 `GiftLevel`
- 杂项数据写入 `ExtraData`

### 6.2 `MemberNickname`

回答的问题：

- “这个粉丝用过哪些昵称？”

它是历史记录，不是波次身份本身。

### 6.3 `MemberAddress`

回答的问题：

- “这个粉丝有哪些地址记录？”
- “哪个地址当前默认可用？”

当前语义应理解为：

- 地址属于全局会员资产
- 允许保留历史、默认、测试地址、软删除状态
- 波次发货记录最终绑定的是某条具体地址记录

### 6.4 `WaveMember`

回答的问题：

- “这个粉丝在这次波次里是什么等级、什么昵称、以什么平台身份参与？”

这是波次快照，不是全局 CRM 主表。

### 6.5 `ProductMaster`

回答的问题：

- “这款商品在系统里的长期主档是什么？”

关键特征：

- 全局商品库
- 当前唯一键为 `(platform, factory_sku)`
- 适合沉淀长期复用的名称、主图、额外信息

### 6.6 `Product`

回答的问题：

- “这款商品在某个波次里实际参与发货时的快照是什么？”

关键特征：

- 波次快照
- 当前唯一键为 `(wave_id, platform, factory_sku)`
- 通过 `ProductMasterID` 关联全局商品主档

### 6.7 `ProductImage` / `ProductMasterImage`

回答的问题：

- “这个波次里这款商品有哪些图？”
- “全局商品主档长期沉淀了哪些图？”

这组分层与 `Product` / `ProductMaster` 的分层保持一致。

### 6.8 `ProductTag`

回答的问题：

- “这款商品应该按哪些规则参与分配？”

当前最重要的两类是：

- `level`
- `user`

### 6.9 `DispatchRecord`

回答的问题：

- “最终该发什么、发给谁、发多少、地址是谁、状态如何？”

如果未来 agent 只记住一个业务事实表，那就应该记住它。

### 6.10 `TemplateConfig`

回答的问题：

- “如何把某个外部平台文件映射到系统内部字段？”

它是输入输出边界，不是核心业务真相。

## 7. 一些关键设计结论

### 7.1 会员等级不应该持久化在全局 `Member`

原因不是“以后绝对用不到”，而是它在当前业务中天然是波次隔离信息：

- 等级会变化
- 它唯一稳定影响的是本次分配策略
- 历史波次需要保留当时事实，而不是被最新等级覆盖

因此，当前分支把它落在 `WaveMember.GiftLevel` 是正确方向。

### 7.2 商品需要“全局主档 + 波次快照”双层结构

这是当前分支已经确定下来的重要架构边界。

只用全局商品库会有问题：

- 一次错误导入会污染所有波次
- 删除波次时容易误删长期资产

只用波次商品也会有问题：

- 同一 SKU 会被重复录入
- 全局图片和商品信息无法沉淀

成熟做法就是：

- 用稳定外部键维护全局主档
- 用波次快照承接当次业务
- 两者通过外键关联，但生命周期不同

### 7.3 `DispatchRecord` 是发货真相，不是派生缓存

它不是“前端列表的一份镜像”，也不是“导出前临时拼一下”的产物。
它就是最终发货明细。

因此任何影响实际履约的改动，都应该问自己：

- 这会如何影响 `DispatchRecord`？
- 需要失效哪些已导出状态？
- 波次状态是否需要重算？

### 7.4 模板系统是必要的，但不是本体

模板系统存在的意义是：

- 隔离各个平台的表头差异
- 降低硬编码列名的成本

但模板系统本身不是“会员回馈代发”这个产品的核心价值。
未来维护时，如果任务与核心业务无关，不要因为模板页面或模板代码复杂，就把所有问题都往模板层归因。

## 8. `柔造` 图片导入的特殊约束

这一条必须单独记录，避免未来维护者误以为“所有平台都能这么匹配图”。

当前实现只对 `柔造` 使用文件名匹配商品图片，原因是：

- 导入时缺少其他足够稳定的关联信息
- 商品平台本身仍然有唯一 SKU，但图片包里未必直接带出可直接用于落库关联的结构化字段

因此当前合理流程是：

1. 在波次商品导入后，先按当前波次商品集做文件名归一化匹配。
2. 将图片去重存入内部资源目录。
3. 生成 `ProductImage`，必要时同步 `ProductMasterImage`。
4. 后续一律依赖内部关联，而不是继续依赖原始文件名。

这条规则的边界也必须明确：

- 这是 `柔造` 当前数据源下的特例
- 不是所有工厂平台的通用规则
- 没有等价外部约束时，不要把文件名匹配推广成通用设计

## 9. 导出语义与“为什么平台字段不在最终订单表里”

这也是一个很容易被误会的点。

导出的订单表，核心上是：

- 会员字段
- 商品字段
- 地址字段
- 数量与状态字段

但它通常不会再带“会员平台”或“商品平台”列，不是因为这些字段在系统里不重要，
而是因为：

- 每次导出本身就是按平台 / 工厂模板分别执行的
- 当前文件天然只服务一个平台
- 工厂真正需要的是它自己的字段顺序和内容，而不是系统内部的归一化维度

内部系统需要保存平台，是为了：

- 唯一识别实体
- 区分导入来源
- 选择模板
- 区分哪一批 `DispatchRecord` 已导出

导出文件不保留平台列，是面向外部目标格式的结果，不代表平台信息在内部不重要。

## 10. 这款软件不是什么

为了避免后续 agent 用错抽象层，下面这些边界也要明确。

它不是：

- 通用电商订单系统
- 库存 / 仓储 / 采购管理系统
- 直播虚拟礼物分析系统
- 会员等级长期画像系统
- 以模板编辑器为核心的低代码平台
- 先有模板、再勉强找业务承载的工具

它首先是：

- 面向创作者会员回馈的实体发货管理工具
- 以“波次分配”和“发货真相”为核心的数据系统

## 11. 用一组问题快速定位该看哪张表

如果未来 agent 不确定某个需求应该落在哪层，可以先问下面这些问题：

- “这个人全局上是谁？”
  - 看 `Member`
- “这个人这次波次是什么等级？”
  - 看 `WaveMember`
- “这款商品的全局主档是什么？”
  - 看 `ProductMaster`
- “这款商品在这次波次里实际使用的是哪个快照？”
  - 看 `Product`
- “这次最终到底发了什么？”
  - 看 `DispatchRecord`
- “这次按等级怎么分？”
  - 看 `ProductTag`
- “这个平台文件怎么映射进来 / 导出去？”
  - 看 `TemplateConfig`

## 12. 给未来维护者和 agent 的明确约束

最后用几条最直接的话收束：

- 先把这款软件理解成“创作者实体会员回馈代发系统”，再去看代码细节。
- `GiftLevel` 是波次身份，不是全局会员属性。
- `ProductMaster` 和 `Product` 不是重复建模，而是全局主档与波次快照的分层。
- `DispatchRecord` 不是可有可无的中间表，而是最终发货真相。
- 模板系统是映射层，不是产品本体。
- `柔造` 的图片文件名匹配是特例，不是通用规则。
- 如果任务与模板无关，就优先在会员、商品、波次、发货记录这几层理解问题。
